# 基于云的协同机械 CAD 系统项目总览与学习准备指南

## 1. 项目定位

本项目的目标是设计并实现一个基于浏览器和云平台的轻量级协同机械 CAD 系统。它不是简单的 Three.js 画图 Demo，也不是传统桌面 CAD 的完整复刻，而是一个面向课程实验的综合型工业软件原型。

从软件架构角度看，本项目至少包含以下几个核心问题：

1. 如何在浏览器中完成基本 2D 草图和 3D 建模交互。
2. 如何用前端图形渲染技术展示机械模型。
3. 如何在后端保存模型数据、项目数据、用户数据和版本数据。
4. 如何实现多人实时协同编辑。
5. 如何将系统部署到华为云或等价云环境中。
6. 如何用文档、演示和答辩证明系统设计合理、功能可用、技术路线可信。

本项目建议采用 MVP 优先策略：先做出一个可演示、可协同、可保存的最小系统，再逐步增强 CAD 能力。

## 2. 软件项目的完整流程概览

一个规范的软件工程项目通常会经历以下阶段：

1. 可行性分析
2. 需求分析
3. 概要设计
4. 技术选型
5. 系统架构设计
6. 详细设计
7. 数据库与数据模型设计
8. 接口设计
9. 原型设计与交互设计
10. 开发环境搭建
11. 编码实现
12. 单元测试与集成测试
13. 联调与性能优化
14. 部署上线
15. 验收、演示与答辩
16. 运维与后续迭代

对于课程实验项目，不一定要把每个阶段都做得像企业级项目一样完整，但这些阶段的核心产物应该尽量体现出来。尤其是架构设计文档、功能需求说明、数据库设计、接口设计、关键模块实现说明、测试说明和部署说明。

## 3. 可行性分析

### 3.1 可行性分析的目的

可行性分析回答的问题是：这个项目能不能做、值不值得做、在当前时间和资源条件下应该做到什么程度。

对本项目来说，完整实现一个工业级 CAD 系统是不现实的。SolidWorks、CATIA、Creo、NX、FreeCAD、Onshape 这类系统都涉及复杂几何内核、约束求解器、工程图、装配、标准件库和大量工业数据格式。课程实验的合理目标应该是实现一个轻量级原型，突出云端协同、浏览器建模、版本管理和基础几何交互。

### 3.2 技术可行性

本项目技术上可行，但需要控制范围。

可行部分：

1. 浏览器端 3D 渲染可以使用 Three.js 或 Babylon.js 实现。
2. 基础几何体、拉伸效果、布尔操作的演示可以用前端几何库或简化模型实现。
3. 2D 草图绘制可以通过 SVG、Canvas 或 Three.js 平面交互实现。
4. 多人协同可以通过 WebSocket 广播操作事件实现。
5. 模型数据可以先用 JSON 格式保存，后续再考虑 B-Rep 或 STEP。
6. 后端用户、项目、文件、版本功能可以用常规 Web 后端实现。
7. 云部署可以使用华为云 ECS、OBS、PostgreSQL、Redis 等服务。

风险较高部分：

1. OpenCascade.js/OCCT Wasm 接入有一定工程复杂度，尤其是 Wasm 体积、加载、内存和 bundler 配置。
2. 真正工业级 B-Rep 建模和布尔运算难度较高。
3. 草图约束求解器难度较高。
4. 真正无冲突的特征树级 CRDT/OT 协同难度较高。
5. DWG、STEP、IGES 等工业格式完整互操作难度较高。

因此建议采用分层实现策略：

1. 最低可交付版本：JSON 特征树 + Three.js 渲染 + 简单 CSG/Cut + WebSocket 协同。
2. 增强版本：优化 CSG 布尔稳定性或引入 OpenCascade.js/OCCT Wasm。
3. 高级版本：引入 Yjs、版本树、参数化约束和格式导入导出。

### 3.3 经济与资源可行性

课程项目通常不追求商业收益，但需要评估资源成本。

主要资源包括：

1. 云服务器：华为云 ECS。
2. 对象存储：华为云 OBS 或本地 MinIO 替代。
3. 数据库：PostgreSQL。
4. 缓存与会话：Redis。
5. 开发人员：3 到 4 人小组。
6. 时间周期：从现在到 6 月 28 日需要高度聚焦 MVP。

资源建议：

1. 开发阶段可以先在本地跑通前后端和 WebSocket。
2. 部署阶段再迁移到华为云 ECS。
3. OBS 如果配置复杂，可以先用本地文件存储抽象接口，后续替换为 OBS SDK。
4. PostgreSQL 可以先本地 Docker 部署，云端再迁移。

### 3.4 时间可行性

建议以课程提交为目标倒排计划。

优先级从高到低：

1. 前端 CAD 工作台界面。
2. 2D 草图基础绘制。
3. 3D 模型展示和简单拉伸。
4. 模型保存与加载。
5. 两人协同同步。
6. 操作历史和版本记录。
7. 云端部署与演示。
8. 文档、视频、答辩材料。

不建议一开始就投入大量时间编译 OCCT Wasm 或深度集成 OpenCascade.js。可以把 OpenCascade.js 技术验证作为技术亮点或增强目标，但 MVP 不应完全依赖它。

### 3.5 法律与合规可行性

需要注意：

1. FreeCAD 是开源项目，参考其架构和实现思想可以，但如果直接使用代码，需要遵守许可证。
2. OpenCASCADE 是开源几何内核，需要确认许可证要求。
3. DWG 格式涉及授权风险，课程项目可仅说明支持计划，不一定实现。
4. AI 生成代码需要人工理解、修改和注明使用方式，不能直接提交未审查代码。

## 4. 需求分析

### 4.1 需求分析的目的

需求分析回答的问题是：系统要为谁解决什么问题，必须做什么，不做什么，做到什么程度才算完成。

本项目的用户可以假设为机械设计团队成员，他们希望在浏览器中打开项目，绘制简单草图，生成三维零件，并与其他成员实时协作。

### 4.2 角色分析

建议定义以下角色：

1. 游客：只能访问登录页或演示页。
2. 普通用户：可以创建项目、编辑模型、保存版本。
3. 项目成员：可以协同编辑指定项目。
4. 项目管理员：可以管理成员权限、删除项目、查看历史版本。
5. 系统管理员：可以管理用户和系统配置，课程项目中可弱化。

### 4.3 核心业务流程

核心业务流程可以设计为：

1. 用户注册或登录。
2. 用户创建 CAD 项目。
3. 用户进入 CAD 工作台。
4. 用户绘制 2D 草图，如矩形、圆、线段。
5. 用户执行拉伸操作生成 3D 模型。
6. 系统在特征树中记录操作。
7. 用户保存模型。
8. 后端保存模型 JSON、操作历史和版本快照。
9. 第二个用户进入同一项目。
10. 两个用户通过 WebSocket 加入同一个协同房间。
11. 用户 A 修改模型，用户 B 的界面实时同步。
12. 系统记录版本，支持撤销、重做或历史回看。

### 4.4 功能性需求

#### 4.4.1 用户与项目管理

最低要求：

1. 用户注册。
2. 用户登录。
3. JWT 鉴权。
4. 创建项目。
5. 查看项目列表。
6. 打开项目。

增强要求：

1. 邀请成员。
2. 设置只读或可编辑权限。
3. 项目归档。
4. 最近打开项目。

#### 4.4.2 CAD 工作台界面

最低要求：

1. 顶部工具栏：选择、直线、矩形、圆、拉伸、切除、布尔并集、布尔差集、保存、撤销、重做。
2. 左侧特征树：显示草图、拉伸、布尔等历史操作。
3. 中央视图区：展示 2D 草图和 3D 模型。
4. 右侧属性面板：显示当前选中对象的参数。
5. 底部状态栏：显示连接状态、当前工具、坐标信息。

增强要求：

1. Ribbon 风格分组工具栏。
2. 多视图切换：前视图、顶视图、等轴测。
3. 网格、吸附、标尺。
4. 快捷键。

#### 4.4.3 2D 草图功能

最低要求：

1. 绘制直线。
2. 绘制矩形。
3. 绘制圆。
4. 选择和移动图元。
5. 修改尺寸参数。

增强要求：

1. 水平约束。
2. 垂直约束。
3. 重合约束。
4. 固定约束。
5. 尺寸驱动。

课程项目中，草图约束可以先做简化实现。例如矩形宽高可修改，圆半径可修改，线段端点可拖拽。

#### 4.4.4 3D 建模功能

最低要求：

1. 将矩形草图拉伸为长方体。
2. 将圆形草图拉伸为圆柱体。
3. 展示模型的旋转、缩放、平移。
4. 模型参数可修改后重新生成。

增强要求：

1. 布尔并集。
2. 布尔差集。
3. 简单切除。
4. 倒角或圆角的视觉模拟。

如果不引入真正几何内核，可以用 Three.js Mesh 表达几何结果，用特征树表达参数化模型。答辩时需要明确说明这是轻量级原型，不是完整工业内核。

#### 4.4.5 文件保存与加载

最低要求：

1. 保存项目模型数据。
2. 加载项目模型数据。
3. 保存操作历史。
4. 记录保存时间、保存人、版本号。

建议模型 JSON 结构包含：

```json
{
  "projectId": "project-001",
  "documentId": "doc-001",
  "version": 3,
  "features": [
    {
      "id": "sketch-001",
      "type": "sketch",
      "entities": []
    },
    {
      "id": "extrude-001",
      "type": "extrude",
      "sourceSketchId": "sketch-001",
      "depth": 40
    }
  ]
}
```

#### 4.4.6 版本管理

最低要求：

1. 每次保存生成一个版本。
2. 可以查看版本列表。
3. 可以加载历史版本。
4. 操作级 Undo/Redo 在前端实现。

增强要求：

1. 类似 Git 的版本树。
2. 分支和合并。
3. 版本差异对比。

MVP 中可以使用线性版本，不必实现复杂版本 DAG。

#### 4.4.7 实时协同

最低要求：

1. 两个用户加入同一项目房间。
2. 显示在线用户列表。
3. 广播用户操作事件。
4. 用户 A 添加或修改图元后，用户 B 自动同步。
5. 显示他人光标位置或选中状态。

增强要求：

1. 引入 Yjs 实现 CRDT。
2. 支持离线编辑后合并。
3. 支持冲突提示。
4. 支持跟随某个用户的视角。

MVP 可以采用操作事件广播：

```json
{
  "type": "feature.added",
  "projectId": "project-001",
  "userId": "user-a",
  "payload": {
    "feature": {
      "id": "extrude-001",
      "type": "extrude",
      "depth": 40
    }
  }
}
```

这种方式比直接同步整个模型更轻量，也更容易解释协同机制。

### 4.5 非功能性需求

#### 4.5.1 性能需求

1. 页面首次加载尽量控制在可接受范围内。
2. 常规模型交互帧率应保持流畅。
3. 协同消息延迟应尽量低，局域或同云区域内建议小于 300ms。
4. 保存和加载项目应有明确反馈。

#### 4.5.2 可用性需求

1. 工具栏操作清晰。
2. 选中对象有高亮反馈。
3. 保存成功、连接断开、冲突发生时有提示。
4. 初次使用者能在 1 分钟内完成绘制矩形并拉伸。

#### 4.5.3 安全需求

1. 登录使用 JWT。
2. 后端接口校验用户身份。
3. 用户只能访问自己有权限的项目。
4. WebSocket 连接需要携带鉴权信息。
5. 文件上传或保存需要限制大小。

#### 4.5.4 可维护性需求

1. 前端组件分层清晰。
2. 后端按模块组织。
3. 模型数据结构稳定。
4. 接口文档清楚。
5. 关键算法和协同流程有注释。

## 5. 概要设计

### 5.1 概要设计的作用

概要设计回答系统整体由哪些模块组成，各模块之间如何协作。

本项目可以划分为：

1. 前端 CAD 工作台。
2. 前端协同客户端。
3. 前端模型状态管理。
4. 后端用户与项目服务。
5. 后端模型文件服务。
6. 后端版本服务。
7. 后端协同 WebSocket 服务。
8. 数据库。
9. 对象存储。
10. 云部署环境。

### 5.2 推荐系统分层

前端分层：

1. UI 层：页面、工具栏、面板、弹窗。
2. 交互层：鼠标事件、拖拽、选择、快捷键。
3. CAD 状态层：草图、特征树、当前工具、选中对象。
4. 几何生成层：根据参数生成 Three.js 几何体。
5. 渲染层：Three.js 场景、相机、光照、网格、控制器。
6. 协同层：WebSocket 连接、消息发送、消息应用。
7. API 层：登录、项目、保存、加载、版本接口。

后端分层：

1. Controller/API 层：接收 HTTP 请求。
2. Service 层：业务逻辑。
3. Repository/DAO 层：数据库访问。
4. WebSocket Gateway：协同连接管理。
5. Storage Adapter：对象存储适配。
6. Auth Module：JWT 鉴权。

### 5.3 推荐技术路线

根据当前团队选择，本项目推荐采用“Vue 前端 + Spring Boot 后端”的技术路线：

1. 前端框架：Vue 3 + TypeScript + Vite。
2. 前端路由：Vue Router。
3. 前端状态管理：Pinia。
4. 3D 渲染：Three.js。
5. 前端 UI 基础组件：Element Plus，可结合自定义 CAD 工作台样式。
6. 后端框架：Spring Boot。
7. HTTP 接口：Spring Web MVC。
8. 鉴权：Spring Security + JWT。
9. 实时协同：Spring WebSocket + STOMP，前端使用 `@stomp/stompjs`。
10. 数据库：PostgreSQL。
11. 数据访问：Spring Data JPA。
12. 数据库迁移：Flyway。
13. 缓存与协同状态：Redis，可先作为可选项，后续用于在线用户、房间状态和短期锁。
14. 云部署：华为云 ECS + Nginx + Docker Compose。
15. 文件存储：开发期本地文件，部署增强期接入华为云 OBS。

这条路线的优势是后端工程化能力强，适合做用户、项目、权限、版本、协同房间等企业级业务模块；前端 Vue 3 上手快，适合快速搭建 CAD 工作台界面。

## 6. 系统架构设计

### 6.1 架构原则

本项目建议采用 Browser-Server 架构，并遵循以下原则：

1. 前端重交互：图形绘制、视角控制、基础几何生成主要放在浏览器。
2. 后端管数据：用户、项目、版本、权限、保存加载由后端负责。
3. 协同走 WebSocket：实时操作同步通过长连接完成。
4. 模型用结构化数据表达：优先保存特征树 JSON，而不是只保存最终 mesh。
5. 几何内核可渐进增强：MVP 先用简化几何生成，后续接入 OpenCascade.js/OCCT Wasm。

### 6.3 关键架构决策

#### 6.3.1 为什么前端承担几何与渲染

CAD 交互对延迟非常敏感。鼠标拖动、旋转视角、修改参数都需要即时反馈。如果每次操作都请求后端生成模型，用户体验会很差。因此基础几何生成和渲染应放在前端。

#### 6.3.2 为什么保存特征树而不是只保存 mesh

mesh 只能表示最终形状，不利于参数化编辑。例如一个长方体如果只保存三角面，后续很难知道它是由哪个矩形草图拉伸而来。特征树可以记录建模过程，例如：

1. Sketch001：矩形，宽 80，高 40。
2. Extrude001：基于 Sketch001 拉伸 30。
3. Cut001：基于 Sketch002 切除 10。

这样才能支持修改参数、撤销、版本对比和协同操作。

#### 6.3.3 为什么 MVP 可以先不用完整 CRDT

真正的 CAD 协同不同于文本协同。文本可以按字符合并，但 CAD 模型涉及特征依赖、几何拓扑和参数引用。课程项目中可以先使用操作事件广播和简单冲突提示。例如同一时间两个用户修改同一特征时，系统提示后到达的操作需要确认或覆盖。

核心后端模块：

1. AuthModule：注册、登录、JWT。
2. UserModule：用户信息。
3. ProjectModule：项目创建、列表、成员。
4. DocumentModule：模型保存和加载。
5. VersionModule：版本快照和历史记录。
6. CollaborationModule：WebSocket 房间和广播。
7. StorageModule：本地文件或 OBS 适配。

### 7.3 协同消息设计

建议定义统一消息格式：

```json
{
  "id": "msg-001",
  "type": "feature.updated",
  "projectId": "project-001",
  "documentId": "doc-001",
  "userId": "user-a",
  "timestamp": 1710000000000,
  "payload": {}
}
```

常见消息类型：

1. room.joined：用户加入房间。
2. room.left：用户离开房间。
3. cursor.moved：远程光标移动。
4. sketch.entity.added：新增草图图元。
5. sketch.entity.updated：更新草图图元。
6. sketch.entity.deleted：删除草图图元。
7. feature.added：新增特征。
8. feature.updated：更新特征。
9. feature.deleted：删除特征。
10. document.saved：文档保存。
11. conflict.detected：冲突提示。

### 7.4 Undo/Redo 设计

建议采用 Command Pattern，即命令模式。

每个操作都封装成命令：

```json
{
  "id": "cmd-001",
  "type": "addRectangle",
  "do": {},
  "undo": {},
  "timestamp": 1710000000000
}
```

例如添加矩形：

1. do：向草图实体列表添加矩形。
2. undo：从草图实体列表移除矩形。

这样可以自然支持：

1. 前端撤销和重做。
2. 操作历史记录。
3. 协同消息广播。
4. 版本保存。

## 10. 原型设计与交互设计

### 10.1 CAD 工作台布局

推荐布局：

1. 顶部：Ribbon 工具栏。
2. 左侧：项目结构和特征树。
3. 中间：3D 视图区。
4. 右侧：属性面板。
5. 底部：状态栏。

这与实验要求中参考 CrownCAD 的方向一致，也符合主流 CAD 软件的认知习惯。

### 10.2 关键交互流程

绘制矩形并拉伸：

1. 用户点击矩形工具。
2. 用户在草图平面上点击并拖拽。
3. 系统生成矩形草图实体。
4. 用户点击拉伸工具。
5. 右侧属性面板输入拉伸深度。
6. 系统生成 3D 模型。
7. 特征树新增 Sketch 和 Extrude 节点。
8. 操作广播给其他在线用户。

保存模型：

1. 用户点击保存。
2. 前端序列化当前 CAD 文档。
3. 调用后端保存接口。
4. 后端生成新版本。
5. 返回版本号。
6. 前端提示保存成功。
7. WebSocket 广播 document.saved。

协同编辑：

1. 用户进入项目后连接 WebSocket。
2. 服务端将用户加入 document room。
3. 用户操作被封装为 operation。
4. operation 先应用到本地状态。
5. operation 发送到服务端。
6. 服务端广播给房间内其他用户。
7. 其他用户接收 operation 并应用到本地状态。

## 17. 风险清单与应对策略

### 17.1 几何内核风险

风险：OpenCascade.js/OCCT Wasm 加载、绑定和前端集成困难。

应对：

1. MVP 先用 Three.js 参数化几何。
2. 把 OpenCascade.js/OCCT Wasm 作为增强功能。
3. 文档中说明未来可替换几何生成层。

### 17.2 协同冲突风险

风险：多人同时编辑同一特征可能导致状态不一致。

应对：

1. MVP 采用最后写入提示或对象级锁。
2. 操作消息带版本号。
3. 检测到版本不一致时提示冲突。
4. 后续升级到 Yjs 或 CRDT。

### 17.3 时间风险

风险：功能范围过大导致都做不完整。

应对：

1. 优先完成可演示闭环。
2. 高级功能放入扩展目标。
3. 每天集成一次。
4. 尽早准备演示脚本。

### 17.4 部署风险

风险：云服务器、Nginx、WebSocket 配置耗时。

应对：

1. 本地先完整跑通。
2. 使用 Docker 简化部署。
3. 提前一周部署到云端。
4. 保留本地演示备用方案。

## 20. 总结

这个项目的本质是一个综合型工业软件原型，关键不在于做出完整商业 CAD，而在于展示清晰的软件工程能力：

1. 能分析项目是否可行。
2. 能明确功能边界。
3. 能设计合理架构。
4. 能实现核心闭环。
5. 能解释关键技术选择。
6. 能完成多人协同和云端存储。
7. 能通过文档、测试和演示证明成果。

后续推进时，建议始终围绕一条主线展开：

用户在浏览器中进入项目，绘制草图，生成三维模型，保存版本，并与另一个用户实时协同。

只要这条主线稳定可演示，项目就有坚实基础。其他高级能力，如 OpenCascade.js/OCCT Wasm、Yjs CRDT、工业级布尔运算、格式导入导出、FEA 仿真，都可以作为加分项逐步加入。
