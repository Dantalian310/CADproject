# 基于云的协同机械 CAD 系统架构设计文档

## 1. 文档说明

### 1.1 文档目的

本文档描述“基于云的协同机械 CAD 系统”的系统架构设计，包括逻辑架构、前端架构、后端架构、数据架构、协同架构、安全架构、部署架构和关键业务链路。

本文档面向后续编码实现，重点说明：

1. 系统各层如何划分。
2. 模块之间如何调用。
3. 前端、后端、数据库、WebSocket 的边界在哪里。
4. 数据如何流动。
5. 协同消息如何传递。
6. 部署时各服务如何连接。
7. 后续扩展点在哪里。

### 1.2 架构目标

系统架构需要满足：

1. 支持浏览器端 CAD 工作台。
2. 支持基础 2D 草图和 3D 建模。
3. 支持模型保存、加载和版本管理。
4. 支持至少两人实时协同编辑。
5. 支持部署到华为云 ECS。
6. 支持后续扩展 OpenCascade.js/OCCT Wasm、OBS、Redis、Yjs 等能力。

### 1.3 架构风格

本项目采用：

```text
Browser-Server 架构
前后端分离架构
模块化单体后端架构
事件驱动协同架构
特征树驱动的轻量 CAD 数据架构
```

说明：

1. 首版不采用微服务，避免部署和调试复杂度过高。
2. 后端采用模块化单体，内部按 auth、project、document、version、collaboration 分模块。
3. 实时协同采用 STOMP over WebSocket。
4. CAD 模型以 JSON 特征树为核心数据结构。

## 3. 前端架构设计

### 3.1 前端架构目标

前端需要完成：

1. CAD 工作台界面。
2. 用户交互。
3. 2D 草图数据维护。
4. 3D 几何渲染。
5. 模型序列化。
6. 协同消息收发和应用。

### 3.2 前端模块架构

```text
pages/
  LoginPage
  ProjectListPage
  WorkspacePage

components/
  toolbar
  feature-tree
  property-panel
  viewport
  status-bar
  collaboration

stores/
  auth.store
  project.store
  cad.store
  collaboration.store

cad/
  model
  sketch
  features
  geometry
  history

api/
  http
  auth.api
  project.api
  document.api
  version.api

collaboration/
  stompClient
  messageTypes
  applyRemoteOperation
```

### 3.3 前端状态流

```text
用户操作
  -> Vue 组件事件
  -> CAD command
  -> cad.store action
  -> CAD 文档状态更新
  -> SceneManager 重建或局部更新
  -> 生成 operation
  -> STOMP 发送给后端
```

### 3.4 前端职责边界

前端负责：

1. CAD 数据结构解释。
2. 草图和特征的交互编辑。
3. Three.js 渲染。
4. Undo/Redo。
5. 本地操作转 operation。
6. 接收远程 operation 并应用。

前端不负责：

1. 用户密码校验。
2. 权限最终判断。
3. 版本号最终生成。
4. 数据库持久化。
5. 云存储上传细节。

### 3.5 Three.js 集成架构

Three.js 不直接散落在各 Vue 组件中，而是封装为：

```text
CadViewport.vue
  -> SceneManager
      -> scene/camera/renderer
      -> controls
      -> mesh registry
      -> raycaster
      -> rebuild(document)
```

设计要求：

1. `CadViewport.vue` 只负责 DOM 容器和用户事件入口。
2. `SceneManager` 负责 Three.js 生命周期。
3. `GeometryKernel` 负责从 CAD 数据生成几何描述。
4. `ThreeGeometryKernel` 是 MVP 实现。

### 3.6 前端扩展点

| 扩展点 | 当前实现 | 后续增强 |
| --- | --- | --- |
| GeometryKernel | Three.js 参数化几何 + 轻量 CSG | OpenCascade.js/OCCT Wasm |
| Collaboration | STOMP 操作广播 | Yjs/CRDT |
| Storage | REST 保存 JSON | OBS 大文件 |
| Sketch Constraint | 固定、水平、垂直、尺寸驱动 | 通用约束求解器 |
| Boolean | 基础 mesh/CSG 并集、差集和 Cut | OCCT 工业级布尔 |

## 4. 后端架构设计

### 4.1 后端架构目标

后端需要完成：

1. 用户认证。
2. 项目管理。
3. 文档保存和加载。
4. 版本管理。
5. 权限校验。
6. WebSocket 协同消息处理。
7. 操作日志记录。
8. 云部署支持。

### 4.2 后端逻辑架构

```text
Controller Layer
  AuthController
  ProjectController
  DocumentController
  VersionController
  CollaborationMessageController

Service Layer
  AuthService
  UserService
  ProjectService
  DocumentService
  VersionService
  OperationService
  SystemEventService
  PresenceService
  StorageService

Repository Layer
  UserRepository
  ProjectRepository
  ProjectMemberRepository
  DocumentRepository
  VersionRepository
  OperationLogRepository
  SystemEventRepository

Infrastructure
  SecurityConfig
  WebSocketConfig
  CorsConfig
  ExceptionHandler
  Flyway Migration
```

### 4.3 后端模块依赖

```text
auth -> user
project -> user
document -> project, version
version -> document, user
collaboration -> document, operation, project
operation -> document, user
system-event -> user, project, document
storage -> independent
security -> user
```

设计要求：

1. `common` 可以被所有模块依赖。
2. 业务模块不要循环依赖。
3. `collaboration` 不直接操作数据库实体细节，复杂逻辑交给 service。
4. `document` 和 `version` 是 CAD 数据持久化核心模块。

### 4.4 事务边界

需要事务的方法：

1. 注册用户。
2. 创建项目并创建默认文档。
3. 保存文档并生成版本。
4. 恢复历史版本并生成新版本。
5. 删除项目或文档，MVP 可不实现。

保存文档事务：

```text
校验权限
  -> 查询当前 document
  -> 计算下一个 versionNumber
  -> 插入 document_versions
  -> 更新 documents.current_version_id
  -> 插入 operation_logs，可选
  -> commit
```

### 4.5 后端扩展点

| 扩展点 | 当前实现 | 后续增强 |
| --- | --- | --- |
| StorageService | JSONB 或本地文件 | OBS |
| PresenceService | 内存 | Redis |
| PermissionService | 项目成员基础权限 | RBAC |
| Collaboration | STOMP 广播 | Redis broker/Yjs |
| Geometry Service | 无 | 云端转换/仿真任务 |

### 5.1 数据分类

| 数据类型 | 示例 | 存储位置 |
| --- | --- | --- |
| 用户数据 | 用户名、邮箱、密码哈希 | PostgreSQL |
| 项目数据 | 项目名、描述、owner | PostgreSQL |
| 权限数据 | 项目成员、角色 | PostgreSQL |
| 文档元数据 | 文档名、当前版本 | PostgreSQL |
| 模型快照 | sketches、features JSON | PostgreSQL JSONB |
| 版本数据 | 版本号、保存人、快照 | PostgreSQL |

| 系统事件 | 登录、WebSocket 连接、权限拒绝、保存失败 | PostgreSQL |
| 在线状态 | 当前在线用户 | 内存/Redis |
| 大文件 | 导出模型、附件 | OBS 可选 |

### 5.2 核心表

```text
users

project_members
documents
document_versions
operation_logs
system_events
```

### 5.3 模型 JSON 架构

```json
{
  "schemaVersion": "1.0",
  "documentId": "doc-001",
  "metadata": {
    "name": "Demo Part",
    "unit": "mm",
    "updatedAt": "2026-06-17T00:00:00.000Z"
  },
  "sketches": [
    {
      "id": "sketch-001",
      "name": "Sketch 1",
      "plane": "XY",
      "entities": []
    }
  ],
  "features": [
    {
      "id": "extrude-001",
      "type": "extrude",
      "sourceSketchId": "sketch-001",
      "sourceEntityId": "rect-001",
      "depth": 30,
      "operation": "new"
    }
  ]
}
```

### 5.4 数据一致性策略

1. 文档保存以版本快照为单位。
2. 每次保存生成新版本，不覆盖历史版本。
3. 当前文档通过 `current_version_id` 指向最新版本。
4. 协同操作可以先实时广播，保存时再固化为版本。
5. 如果客户端版本落后，保存时后端返回冲突提示。

## 6. 协同架构设计

### 6.1 协同目标

MVP 需要实现：

1. 两个用户打开同一文档。
2. 在线用户列表更新。
3. 用户 A 操作同步到用户 B。
4. 远程光标实时显示。
5. 简单冲突提示。
6. 服务端维护文档房间连接池或等价 session 映射。
7. 广播增量 operation，而不是每次广播完整模型。

### 6.2 协同通信架构

```text
Client A
  STOMP SEND /app/documents/{id}/operations
        |
        v
Spring CollaborationMessageController
        |
        +-> OperationService 记录日志
        |
        v
  BROADCAST /topic/documents/{id}/operations
        |
        +-> Client B
        +-> Client C
```

### 6.3 Topic 设计

| Destination | 用途 |
| --- | --- |
| `/app/documents/{id}/join` | 加入文档协同 |
| `/app/documents/{id}/leave` | 离开文档协同 |
| `/app/documents/{id}/operations` | 发送 CAD 操作 |
| `/app/documents/{id}/cursor` | 发送光标 |
| `/topic/documents/{id}/presence` | 在线用户广播 |
| `/topic/documents/{id}/operations` | 操作广播 |
| `/topic/documents/{id}/cursor` | 光标广播 |
| `/topic/documents/{id}/system` | 系统提示和冲突提示 |

### 6.4 Operation 消息格式

```json
{
  "id": "op-001",
  "type": "sketch.entity.added",
  "documentId": "doc-001",
  "userId": "user-001",
  "baseVersion": 3,
  "targetId": "rect-001",
  "timestamp": 1710000000000,
  "payload": {
    "entity": {
      "id": "rect-001",
      "type": "rectangle",
      "x": 0,
      "y": 0,
      "width": 80,
      "height": 40
    }
  }
}
```

### 6.5 冲突处理策略

MVP 采用简单策略：

1. operation 携带 `baseVersion`。
2. 如果客户端版本落后，显示提示。
3. 同一对象被多人同时编辑时，后到操作提示冲突。
4. 首版可采用“后写覆盖 + 提示”，P1 再加入对象级锁。

不在 MVP 中实现：

1. 完整 CRDT。
2. 复杂特征树合并。
3. 离线编辑自动合并。

## 7. 安全架构设计

### 7.1 安全目标

1. 未登录用户不能访问受保护 API。
2. 用户只能访问自己有权限的项目和文档。
3. 密码不能明文存储。
4. WebSocket 连接需要鉴权。
5. 云端密钥不能写入代码仓库。

### 7.2 HTTP 鉴权

```text
前端登录
  -> 后端返回 JWT
  -> 前端保存 token
  -> Axios 请求附加 Authorization: Bearer token
  -> JwtAuthenticationFilter 校验
  -> Controller 执行业务
```

### 7.3 WebSocket 鉴权

方案：

1. 前端连接 STOMP 时在 header 中携带 token。
2. 后端在 WebSocket handshake 或 channel interceptor 中校验 token。
3. 校验通过后绑定用户身份。
4. 发送 operation 时再次校验 document 权限。

### 7.4 权限模型

MVP 权限：

```text
OWNER：项目创建者，可读写
EDITOR：项目成员，可读写
VIEWER：只读，可选
```

首版可以只实现 OWNER 和 EDITOR。

权限分配入口：

1. OWNER 可以通过项目成员接口添加成员。
2. OWNER 可以将成员设置为 EDITOR 或 VIEWER。
3. EDITOR 可以编辑模型，但不能分配权限。
4. 后端在项目、文档、版本和协同房间入口统一校验权限。

### 7.5 安全边界

前端权限控制只用于体验，后端必须做最终判断。

必须后端校验：

1. 查询项目。
2. 打开文档。
3. 保存文档。
4. 查询版本。
5. 恢复版本。
6. 加入 WebSocket 文档房间。

   v

### 8.2 部署组件

| PostgreSQL | Docker 容器 | 持久化 volume |
| Redis | Docker 容器，可选 | 后续协同状态 |

### 8.3 环境变量

后端：

```text
DATABASE_URL
DATABASE_USERNAME
DATABASE_PASSWORD
JWT_SECRET
JWT_EXPIRES_IN
CORS_ALLOWED_ORIGIN
REDIS_HOST
REDIS_PORT
```

```text

```

### 8.4 部署原则

1. 数据库端口不对公网开放。

## 9. 关键业务时序

### 9.1 登录时序

```text
User -> LoginPage: 输入账号密码
LoginPage -> Auth API: POST /api/auth/login
AuthController -> AuthService: 校验用户
AuthService -> UserRepository: 查询用户
AuthService -> PasswordEncoder: 校验密码
AuthService -> JWT: 生成 token
AuthController -> LoginPage: 返回 token
LoginPage -> Router: 跳转项目列表
```

### 9.2 保存文档时序

```text
User -> WorkspacePage: 点击保存
WorkspacePage -> cad.store: serialize()
WorkspacePage -> Document API: PUT /api/documents/{id}/save
DocumentController -> DocumentService: saveDocument
DocumentService -> PermissionService: 校验权限
DocumentService -> VersionService: createVersion
VersionService -> VersionRepository: insert document_version
DocumentService -> DocumentRepository: update current_version_id
DocumentService -> OperationLogRepository: 记录保存操作
DocumentController -> WorkspacePage: 返回新版本号
WorkspacePage -> cad.store: dirty=false
```

### 9.3 协同操作时序

```text
User A -> WorkspacePage A: 绘制矩形
WorkspacePage A -> cad.store: apply local command
WorkspacePage A -> STOMP: send operation
STOMP -> CollaborationMessageController: receive
CollaborationMessageController -> OperationService: log operation
CollaborationMessageController -> Broker: broadcast topic
Broker -> WorkspacePage B: receive operation
WorkspacePage B -> cad.store: apply remote operation
WorkspacePage B -> SceneManager: rebuild
```

### 9.4 加载历史版本时序

```text
User -> VersionDialog: 选择版本
VersionDialog -> Version API: GET version detail
VersionController -> VersionService: getVersion
VersionService -> PermissionService: 校验权限
VersionService -> VersionRepository: 查询快照
VersionController -> VersionDialog: 返回 snapshotJson
VersionDialog -> cad.store: loadDocument(snapshot)
VersionDialog -> SceneManager: rebuild
```

## 10. 性能架构考虑

### 10.1 前端性能

策略：

1. Three.js 对象复用或统一释放。
2. CAD 状态变化后优先局部更新，MVP 可先整体 rebuild。
3. 避免把大型 Three.js 对象放入 Pinia。
4. 大量草图图元时再做空间索引或分层渲染。

### 10.2 后端性能

策略：

1. 保存文档按版本快照写入。
2. operation 日志可以异步优化，MVP 同步写入也可接受。
3. 版本列表只返回元数据，不返回完整 JSON。
4. 模型快照较大后迁移到 OBS。

### 10.3 协同性能

策略：

1. 光标消息节流。
2. 高频拖拽操作可合并。
3. 操作消息只传增量，不传完整模型。
4. 多实例部署时引入 Redis 或消息代理。

### 10.4 可观测性与运行状态

参考 Rapid SCADA 这类工业软件平台的工程思想，系统不只要“能运行”，还要能在演示和部署时解释当前运行状态。

后端需要暴露和记录：

1. `/actuator/health`：用于 ECS/Nginx 部署后的健康检查。
2. `operation_logs`：记录 CAD 操作、保存、撤销、重做和协同广播。
3. `system_events`：记录登录、WebSocket 连接/断开、房间加入/离开、权限拒绝和保存失败等系统事件。
4. 结构化日志字段至少包含 `traceId`、`userId`、`projectId`、`documentId`、`eventType`、`result`。

前端需要展示：

1. API 连接状态。
2. WebSocket 连接状态。
3. 当前保存状态。
4. 在线协作者数量。
5. 最近一次冲突提示或同步错误。

编码落点：

1. 后端增加 `SystemEventService`，由认证、文档、协同模块调用。
2. 前端增加 `statusStore`，统一维护连接、保存和同步状态。
3. CAD 工作台底部状态栏只读取 `statusStore`，不直接散落各组件状态。
4. 运维类接口仅在开发和答辩环境开放，生产环境要限制敏感信息。

## 11. 可扩展架构

### 11.1 几何内核扩展

当前：

```text
ThreeGeometryKernel
```

未来：

```text
OpenCascadeJsGeometryKernel
```

要求：

1. 上层 CAD store 不依赖具体几何实现。
2. 几何层输入统一为 CadDocument。
3. 输出统一为渲染描述、Three.js mesh 数据或可导入的 GLB/glTF 数据。
4. OpenCascade.js 技术验证先在独立 demo 完成 Box - Sphere Cut，再决定是否进入主线。

### 11.2 存储扩展

当前：

```text
PostgreSQL JSONB
```

未来：

```text
OBS Object Storage
```

要求：

1. DocumentVersion 保留 `snapshotPath`。
2. StorageService 抽象本地、数据库、OBS 差异。

### 11.3 协同扩展

当前：

```text
STOMP 操作广播
```

未来：

```text
Yjs/CRDT
Redis Pub/Sub
多实例 WebSocket
```

要求：

1. operation 类型稳定。
2. 客户端应用远程操作的入口统一。
3. 服务端协同模块与文档保存模块解耦。

## 12. 架构风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| Three.js 无法完成复杂 CAD 建模 | 专业性不足 | 明确 MVP 边界，预留 OCCT |
| WebSocket 鉴权复杂 | 安全漏洞 | 使用 channel interceptor 校验 token |
| JPA JSONB 映射不稳定 | 保存失败 | 先用字符串/JsonNode，测试后优化 |
| 协同冲突复杂 | 状态不一致 | MVP 做操作广播和冲突提示 |
| 云端部署 WebSocket 失败 | 无法演示协同 | 提前验证 Nginx Upgrade 配置 |
| 文档模型频繁变更 | 前后端不一致 | 固定 schemaVersion，变更时升级版本 |

## 13. 架构落地顺序

建议按以下顺序实现：

1. 建立前后端工程。
2. 建立数据库和 Flyway 初始表。
3. 实现登录和 JWT。
4. 实现项目、成员权限和文档接口。
5. 实现 CAD 工作台布局。
6. 实现 Three.js 基础视图。
7. 实现 CAD 数据模型和 Pinia store。
8. 实现直线、矩形、圆、基础约束和尺寸驱动。
9. 实现拉伸、Cut、布尔并集和差集。
10. 实现保存、加载、版本和 Undo/Redo。
11. 实现 STOMP 连接、连接映射和增量操作广播。
12. 实现在线用户、远程光标和冲突提示。
13. 补充椭圆、多段线、网格吸附等低难度绘图增强。
14. 部署到 ECS。
15. 补充测试和演示材料。

## 14. 架构设计结论

本系统采用以浏览器端 CAD 交互为核心、Spring Boot 后端业务服务为支撑、PostgreSQL 版本化存储为基础、WebSocket/STOMP 为实时协同通道的整体架构。

该架构的关键优点：

1. 结构清晰，适合小组协作开发。
2. 技术栈成熟，便于课程周期内落地。
3. 主链路明确，便于演示和答辩。
4. 保留了几何内核、云存储、协同算法的后续扩展空间。

后续编码必须围绕“模型数据结构、CAD 操作命令、版本快照、协同 operation”四个核心概念展开。
