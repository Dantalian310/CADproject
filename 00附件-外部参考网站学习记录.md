本次参考的网站包括：

## 2. FreeCAD 参考结论

### 2.1 参考价值

FreeCAD 是开源参数化 3D CAD 项目，官方源码说明其核心特点包括：

1. 参数化 3D 建模。
2. 从 2D 草图生成 3D 模型。
3. 通过模型历史修改参数。
4. 基于 OpenCASCADE 作为几何内核。
5. 模块化结构，支持不同工作台。

### 2.2 对本项目的启发

本项目应继续坚持：

1. 使用特征树记录建模历史。
2. 保存参数化模型 JSON，而不是只保存 mesh。
3. 将草图、特征、几何生成、渲染分层。
4. 后续接入 OCCT/OpenCascade.js 时，只替换 GeometryKernel，不推翻上层 CAD 数据结构。

### 2.3 融入文档位置

1. `03-概要设计文档.md`：强化特征树和 GeometryKernel 抽象。
2. `05-系统架构设计文档.md`：强化参数化特征树数据架构。
3. `04附件-技术栈说明文档.md`：FreeCAD 继续作为参考项目。

## 3. Rapid SCADA 参考结论

### 3.1 参考价值

Rapid SCADA 不是 CAD 软件，但它是工业软件平台，具备以下值得参考的特点：

1. 开源工业自动化平台。
2. 支持 Web 访问。
3. 支持服务器、嵌入式设备和云端运行。
4. 支持分布式节点之间实时交换信息。
5. 通过模块扩展基础功能。
6. 强调日志、可靠性、技术支持和工业现场可运维性。

### 3.2 对本项目的启发

虽然本项目是 CAD，不是 SCADA，但可以吸收工业软件工程思想：

1. 后端模块化单体要保留清晰扩展点。
2. 协同房间、在线用户、操作日志应作为可观测对象。
3. 后续部署应加入健康检查、日志和运行状态页。
4. 可以设计 `OperationLog` 和 `SystemEvent` 作为调试与演示依据。
5. 后续可在 CAD 工作台底部状态栏展示连接状态、保存状态、协同状态。

### 3.3 融入文档位置

1. `05-系统架构设计文档.md`：增加可观测性和工业软件运行状态设计。
2. `10-开发环境搭建与实施前准备清单.md`：保留 Actuator、日志和健康检查准备。

## 4. OpenCascade.js 参考结论

### 4.1 参考价值

OpenCascade.js 是将 OpenCascade CAD 库通过 Emscripten 移植到 JavaScript/WebAssembly 的项目。官方文档强调：

1. 可以构建运行在浏览器、云端或支持 WebAssembly 设备上的 CAD 应用。
2. 使用 Emscripten 和 WebAssembly 让 CAD Kernel 接近原生速度。
3. 支持按需自定义构建以降低体积。
4. 示例中展示了使用 OpenCascade.js 创建形体、执行 Boolean Cut，并导出 GLB 进行 Web 可视化。
5. 官方提供 starter templates，说明浏览器中配置 Emscripten 产物和 bundler 有一定复杂度。
6. 官方文档明确 `.wasm` 资产体积较大，需要依赖浏览器缓存、gzip/brotli 压缩或自定义构建降低加载成本。
7. 官方 starter templates 已包含浏览器和服务端方向，其中浏览器方向列出 React/Vue 相关模板，可作为 Vue/Vite 技术验证参考。

### 4.2 对本项目的启发

OpenCascade.js 对本项目非常有参考价值，但不建议直接作为 MVP 强依赖。

建议定位：

1. MVP 继续使用 Three.js + three-bvh-csg 实现课程级 Cut/Boolean。
2. P1/P2 阶段增加 `OpenCascade.js 技术验证`。
3. 设计 `OpenCascadeJsGeometryKernel` 作为未来替代 `ThreeGeometryKernel` 的实现。
4. 技术验证目标可以非常小：Box - Sphere Cut -> GLB -> Three.js/model-viewer 显示。
5. 若技术验证成功，再逐步将拉伸、布尔、STEP/glTF 导出迁移到 OpenCascade.js。

### 4.3 需要注意的风险

1. `opencascade.js@beta` 说明其生态仍需谨慎评估。
2. Wasm 包体积、加载时间和内存占用需要单独测试。
3. Vite/Vue bundler 配置可能需要额外处理 Emscripten 生成的 JS/Wasm 文件。
4. OpenCascade.js API 接近 OCCT C++ 风格，学习成本高。
5. 需要清晰封装 GeometryKernel，避免业务代码直接依赖大量 OCCT API。
6. 若引入主线，需要把 Wasm 文件缓存策略、加载进度提示和失败回退设计进前端体验。

### 4.4 融入文档位置

1. `04-技术选型文档.md`：将 OpenCascade.js 作为 OCCT Web 端接入候选方案。
2. `04附件-技术栈说明文档.md`：补充 OpenCascade.js 的作用、优势和难点。
3. `05-系统架构设计文档.md`：增加 `OpenCascadeJsGeometryKernel` 扩展路径。
4. `10-开发环境搭建与实施前准备清单.md`：增加 OpenCascade.js 技术验证准备。

## 5. 总体吸收结论

本次外部资料对项目规划有实际帮助。

吸收结果：

1. FreeCAD 强化参数化特征树和工作台模块化思路。
2. Rapid SCADA 强化工业软件可观测性、日志、状态监控和模块化扩展思路。
3. OpenCascade.js 将原来笼统的 “OCCT Wasm 后续增强” 具体化为可执行的 Web 端技术验证路径。

项目主线不变：

```text
Vue 3 + Three.js + Spring Boot + PostgreSQL + STOMP
```

技术增强路径更新为：

```text
MVP：Three.js + three-bvh-csg
P1/P2：OpenCascade.js 技术验证
高级阶段：OpenCascade.js/OCCT 驱动专业几何内核
```
