# 基于云的协同机械 CAD 系统可行性分析文档

## 1. 文档说明

### 1.1 文档目的

本文档用于分析“基于云的协同机械 CAD 系统设计与实现”项目在技术、时间、资源、成本、运维、合规和风险控制等方面的可行性，并给出适合课程实验场景的项目边界和实现策略。

本文档的核心结论是：本项目具备实现轻量级原型系统的可行性，但不具备在短周期内实现完整工业级 CAD 系统的现实条件。因此，项目应采用“最小可用闭环优先，复杂 CAD 能力渐进增强”的路线。

### 1.2 项目背景

传统机械 CAD 软件通常以桌面端为主，存在安装维护复杂、协同能力弱、数据版本管理困难、跨平台访问成本高等问题。随着工业软件云化和 Web 图形技术的发展，基于浏览器的 SaaS 化 CAD 系统成为重要方向。

本项目要求以云平台为基础，参考 FreeCAD 的开源 CAD 架构和 CrownCAD 的交互体验，实现一个具备 2D 草图、简单 3D 建模、文件持久化、版本管理和多人实时协同能力的轻量级机械 CAD 系统。

### 1.3 项目目标

本项目目标不是复刻 SolidWorks、CATIA、Creo、NX、FreeCAD 或 Onshape 的完整能力，而是实现一个可演示、可解释、可部署的协同 CAD 原型。

项目最低目标包括：

1. 用户可以在浏览器中进入 CAD 工作台。
2. 用户可以绘制基本 2D 草图，如直线、矩形、圆。
3. 用户可以基于简单草图生成 3D 模型，如长方体、圆柱体或简单拉伸体。
4. 系统可以保存和加载模型数据。
5. 系统可以记录模型版本。
6. 至少两个用户可以同时打开同一模型并实时同步操作。
7. 系统可以部署到华为云或等价云环境。
8. 项目可以通过实验报告、架构图、演示视频和答辩说明其设计合理性。

## 2. 可行性结论摘要

### 2.1 总体结论

项目可行，但必须控制实现范围。

推荐实现范围：

1. 前端采用 Vue 3 + TypeScript + Vite。
2. 3D 渲染采用 Three.js。
3. CAD 数据采用结构化 JSON 特征树保存。
4. 后端采用 Spring Boot，配合 Spring Web MVC、Spring Security、Spring WebSocket。
5. 数据库采用 PostgreSQL。
6. 数据访问采用 Spring Data JPA，数据库迁移采用 Flyway。
7. 实时协同采用 Spring WebSocket + STOMP，MVP 阶段采用操作事件广播。
8. 部署采用华为云 ECS + Docker Compose + Nginx。
9. OBS、RDS、Redis、CCE 作为增强项或部署优化项。

不建议首版强依赖：

1. 完整 OpenCascade.js/OCCT Wasm 几何内核。
2. 完整草图约束求解器。
3. 完整 CRDT/OT 特征树合并。
4. STEP/IGES/DWG 等工业格式完整导入导出。
5. 工程图、装配、FEA 仿真等高级 CAD 模块。

### 2.2 MVP 建议边界

MVP 应围绕一条核心演示链路：

用户登录系统，创建项目，进入 CAD 工作台，绘制矩形或圆，执行拉伸生成 3D 模型，保存版本，另一个用户在同一项目中看到实时同步结果。

只要这条链路稳定，项目就能覆盖课程实验的主要评分点：架构设计、核心功能实现、实时协同、云端存储、代码质量和演示答辩。

## 3. 技术可行性分析

### 3.1 浏览器端 CAD 工作台可行性

浏览器端实现 CAD 工作台是可行的。

原因如下：

1. 现代浏览器支持 WebGL，可以承载 3D 模型渲染。
2. Three.js 对 WebGL 做了良好封装，可以快速实现相机、光照、网格、几何体、材质、选中和交互。
3. Vue 3 可以用单文件组件组织复杂的工具栏、特征树、属性面板和状态栏。
4. TypeScript 可以提高前端复杂 CAD 状态结构的维护性。
5. 对课程项目来说，简单 2D 草图和参数化 3D 几何可以用结构化数据和 Three.js Geometry 实现。

可实现能力：

1. 绘制线段、矩形、圆。
2. 选择、移动、删除图元。
3. 修改图元参数。
4. 从草图生成拉伸模型。
5. 显示坐标网格和辅助坐标轴。
6. OrbitControls 实现旋转、缩放和平移。

主要风险：

1. 鼠标坐标到模型坐标转换需要处理好。
2. 2D 草图和 3D 视图之间的状态切换需要设计清楚。
3. 选中、拖拽、吸附、尺寸输入等交互容易消耗时间。

应对策略：

1. 先使用固定 XY 平面作为草图平面。
2. 先支持矩形和圆，不急于支持复杂自由曲线。
3. 先做参数化几何体，不追求真实工业级拓扑。
4. 每一种操作都记录为 Feature，便于撤销、保存和协同。

### 3.2 3D 建模能力可行性

简单 3D 建模可行，完整工业建模不可在短周期内完成。

MVP 可实现方式：

1. 矩形草图拉伸为 BoxGeometry 或 ExtrudeGeometry。
2. 圆形草图拉伸为 CylinderGeometry。
3. 简单多边形草图使用 Three.js Shape + ExtrudeGeometry。
4. 简单 Cut、布尔并集和差集可以引入轻量 CSG 库实现课程级演示。

不建议首版实现完整 B-Rep：

1. B-Rep 数据结构复杂。
2. 拓扑命名和特征依赖复杂。
3. 工业级布尔运算、倒角、圆角、抽壳等操作依赖成熟几何内核。
4. OpenCascade.js 虽然提供了更直接的 OCCT WebAssembly 封装，但 Wasm 加载、内存占用、Vite/Vue 配置和 API 学习仍需要额外工程经验。

建议结论：

1. MVP 阶段使用 Three.js 参数化几何生成。
2. 架构上预留 GeometryKernel 接口。
3. 增强阶段再替换或接入 OpenCascade.js/OCCT Wasm。

示例架构抽象：

```ts
interface GeometryKernel {
  createBox(params: BoxParams): MeshData
  createCylinder(params: CylinderParams): MeshData
  extrudeSketch(sketch: Sketch, params: ExtrudeParams): MeshData
}
```

这样即使 MVP 使用 Three.js，后续也可以把 GeometryKernel 的实现替换为 OpenCascade.js/OCCT Wasm。

### 3.3 草图约束求解可行性

简化草图约束可行，完整约束求解器风险较高。

可实现的简化能力：

1. 矩形保持水平和垂直。
2. 圆保持圆心和半径。
3. 线段端点可以拖拽。
4. 属性面板修改宽、高、半径、长度。
5. 简单固定约束可以表现为禁止拖动。

复杂约束求解器风险：

1. 需要构建约束方程。
2. 需要数值求解器。
3. 需要处理过约束和欠约束。
4. 需要用户交互反馈。
5. 与 CAD 拓扑依赖耦合较强。

建议结论：

1. MVP 不实现通用约束求解器。
2. 先实现“参数驱动”的简化约束。
3. 文档中说明后续可引入 dogleg、Levenberg-Marquardt 或基于 Eigen 的求解器。

### 3.4 后端服务可行性

后端功能具备较高可行性。

推荐后端功能：

1. 用户注册登录。
2. JWT 鉴权。
3. 项目创建和项目列表。
4. CAD 文档保存和加载。
5. 版本快照保存。
6. WebSocket 房间管理。
7. 在线用户列表。
8. 协同操作广播。

推荐技术：

1. Spring Boot：工程化成熟，适合用户、项目、权限、版本等后端业务模块。
2. Spring Web MVC：用于 REST API。
3. Spring Security + JWT：用于登录鉴权和接口保护。
4. Spring WebSocket + STOMP：用于协同房间、主题订阅和操作广播。
5. Spring Data JPA：用于用户、项目、文档、版本等实体持久化。
6. PostgreSQL：适合存储用户、项目、版本元数据，也支持 JSONB 保存模型快照。
7. Flyway：用于数据库表结构迁移和版本化。

主要风险：

1. 鉴权和 WebSocket 连接鉴权容易遗漏。
2. 前后端模型 JSON 结构必须统一。
3. 并发保存时需要处理版本号。

应对策略：

1. 所有模型变更统一走 operation 结构。
2. 保存接口携带 baseVersion。
3. 服务端保存时检查版本是否落后。
4. WebSocket 加入房间前校验 JWT 和项目权限。

### 3.5 多人实时协同可行性

两人实时协同可行，完整无冲突协同难度较高。

MVP 推荐机制：

1. 用户进入同一 CAD 文档时加入同一个 WebSocket room。
2. 用户操作被封装为 operation。
3. 本地先应用 operation。
4. operation 发送到服务端。
5. 服务端广播给房间内其他客户端。
6. 其他客户端接收后应用同样的 operation。

示例：

```json
{
  "type": "feature.added",
  "documentId": "doc-001",
  "userId": "user-a",
  "baseVersion": 3,
  "payload": {
    "feature": {
      "id": "extrude-001",
      "type": "extrude",
      "sourceSketchId": "sketch-001",
      "depth": 40
    }
  }
}
```

可实现协同能力：

1. 在线用户列表。
2. 他人光标位置。
3. 草图新增同步。
4. 特征新增同步。
5. 参数修改同步。
6. 保存状态同步。
7. 简单冲突提示。

不建议首版实现完整 CRDT：

1. CAD 特征树不是纯文本。
2. 操作之间有依赖关系。
3. 几何拓扑修改很难自动合并。
4. Yjs 更适合作为后续升级方案，而不是第一版强依赖。

建议结论：

1. MVP 使用 WebSocket 操作广播。
2. 增强版引入 Yjs 管理共享文档。
3. 对同一对象并发修改，MVP 使用对象级锁或冲突提示。

### 3.6 云部署可行性

部署到华为云 ECS 可行。

推荐部署路径：

1. 本地开发完成前后端。
2. 使用 Docker Compose 在本地跑通 frontend、backend、postgres、redis。
3. 购买或申请华为云 ECS。
4. 在 ECS 上安装 Docker 和 Nginx。
5. 通过 Docker Compose 部署服务。
6. Nginx 反向代理前端、API 和 WebSocket。
7. 可选接入 OBS 保存模型快照文件。
8. 可选使用 RDS PostgreSQL 替代容器内 PostgreSQL。

MVP 部署方案：

1. 前端构建为静态文件，由 Nginx 托管。
2. 后端 Node 服务运行在 3000 端口。
3. PostgreSQL 可以先用 Docker 容器。
4. Redis 可选。
5. 安全组只开放 22、80、443，后端端口只在内网访问。

增强部署方案：

1. RDS PostgreSQL 托管数据库。
2. OBS 存储模型快照。
3. Redis 云服务管理会话或协同锁。
4. CCE/Kubernetes 用于容器集群部署。

课程项目不建议一开始上 CCE。Kubernetes 能力强，但配置成本较高，容易分散开发注意力。

## 4. 资源可行性分析

### 4.1 人员资源

建议团队 3 到 4 人。

推荐分工：

1. 架构与后端负责人：后端架构、数据库、接口、鉴权、版本管理。
2. 前端工作台负责人：页面布局、工具栏、特征树、属性面板。
3. 图形建模负责人：Three.js、草图交互、拉伸建模、选中与拖拽。
4. 协同与部署负责人：WebSocket、在线用户、云部署、演示环境。

如果只有 2 人：

1. 一人负责前端 CAD 与图形。
2. 一人负责后端、协同、部署。

如果只有 1 人：

1. 必须压缩功能范围。
2. 用户管理可做简化。
3. 云部署可只做 ECS 单机部署。
4. 协同只保证两个浏览器窗口同步。

### 4.2 时间资源

以 6 月 28 日提交为约束，时间非常紧张，因此应采用迭代交付。

建议倒排：

1. 第 1-2 天：完成文档、仓库、技术选型、原型设计。
2. 第 3-5 天：完成前后端脚手架、登录、项目列表。
3. 第 6-9 天：完成 CAD 工作台、Three.js 视图、基础草图。
4. 第 10-12 天：完成拉伸、特征树、保存加载。
5. 第 13-14 天：完成 WebSocket 协同和在线用户。
6. 第 15 天：部署到云端。
7. 第 16 天：测试、修复、录制演示、完善报告。

如果中途进度落后，优先保留：

1. CAD 工作台。
2. 简单建模。
3. 保存加载。
4. 两人协同。

可以砍掉：

1. 复杂约束。
2. 工业级复杂布尔、倒角和圆角。
3. OBS。
4. RDS。
5. Redis。
6. CCE。
7. 工业格式导入导出。

### 4.3 硬件和云资源

本地开发建议：

1. Windows 10/11 或 Linux/macOS。
2. 16GB 内存更稳，8GB 也可完成 MVP。
3. Chrome 或 Edge 浏览器。
4. Docker Desktop 可选但推荐。

云端建议：

1. ECS：2 vCPU、4GB 内存即可支撑课程演示。
2. 系统：Ubuntu 22.04 LTS 或 24.04 LTS。
3. 存储：40GB 系统盘。
4. 带宽：1-5 Mbps 可用于课程演示。
5. 数据库：MVP 可用 Docker PostgreSQL，增强版用 RDS。
6. 对象存储：MVP 可用本地文件，增强版用 OBS。

## 5. 经济可行性分析

本项目为课程实验项目，经济成本应尽量低。

可能成本：

1. 华为云 ECS 费用。
2. 云数据库 RDS 费用。
3. OBS 存储和请求费用。
4. 公网带宽费用。
5. 域名和 HTTPS 证书费用，可选。

降低成本策略：

1. 开发阶段全部本地运行。
2. 云端只在最后部署和演示时使用。
3. 数据库先使用 Docker 容器。
4. OBS 作为可选增强。
5. 使用免费 SSL 证书或直接 HTTP 演示。
6. 及时释放不用的云资源。
7. 设置预算告警。

结论：在课程实验规模下，经济成本可控。

## 6. 操作可行性分析

### 6.1 用户操作可行性

系统面向学生实验和演示用户，操作流程应简单。

最低操作链路：

1. 登录。
2. 创建项目。
3. 进入 CAD 工作台。
4. 点击矩形工具。
5. 绘制矩形。
6. 点击拉伸。
7. 输入深度。
8. 保存。
9. 打开另一个浏览器窗口查看同步。

操作设计原则：

1. 工具栏按钮清晰。
2. 当前工具状态明确。
3. 选中对象有高亮。
4. 保存和连接状态有提示。
5. 失败时给出可理解错误信息。

### 6.2 开发操作可行性

为降低团队协作难度，应采用统一开发规范：

1. 统一 Node 版本。
2. 统一包管理器。
3. 统一代码格式化工具。
4. 统一分支策略。
5. 统一提交信息格式。
6. 统一环境变量模板。

建议统一工具：

1. Node.js LTS。
2. pnpm。
3. ESLint。
4. Prettier。
5. GitHub 或 Gitee。
6. Docker Compose。

## 7. 法律与合规可行性分析

### 7.1 开源许可证

需要注意：

1. FreeCAD 采用 LGPL 许可证，参考其架构思想没有问题，但直接复制代码需要遵守许可证。
2. OpenCASCADE 采用开源许可证，可用于 CAD/CAM/CAE 软件开发，但需要保留许可证声明。
3. Three.js、Vue、Spring Boot、Yjs 等主流开源库通常允许课程项目使用，但仍需在文档中列明依赖。
4. 如果使用第三方 CSG 库，需要检查其许可证。

建议：

1. 本项目尽量不直接复制 FreeCAD 源码。
2. 将 FreeCAD 作为架构参考和概念学习对象。
3. 如引入第三方库，在 README 或报告中列出名称、用途、许可证和链接。

### 7.2 工业格式合规

DWG 等格式涉及商业授权和生态限制。课程项目不建议实现 DWG 读写。

建议：

1. MVP 使用项目自定义 JSON 模型格式。
2. 演示导出可支持 JSON。
3. STL、glTF 可以作为后续增强项。
4. STEP/IGES 需要依赖 OpenCASCADE 或其他库，作为后续计划说明即可。

### 7.3 AI 工具使用合规

实验要求允许使用 AI 编程助手，但必须注明使用方式。

建议：

1. 在报告中单独写“AI 工具使用说明”。
2. 说明 AI 用于架构讨论、代码草稿、报错排查、文档润色。
3. 明确最终代码经过人工理解、修改和测试。
4. 不提交未经审查的 AI 原样代码。

## 8. 风险分析与应对策略

### 8.1 技术风险

| 风险 | 等级 | 影响 | 应对 |
| --- | --- | --- | --- |
| OpenCascade.js/OCCT Wasm 难以集成 | 高 | 影响真实几何能力 | MVP 不强依赖，预留接口并先做独立 demo |
| 草图约束求解复杂 | 高 | 影响 CAD 专业性 | 先做参数驱动和简化约束 |
| 协同冲突难处理 | 高 | 影响多人编辑稳定性 | 采用操作广播 + 对象级锁 + 冲突提示 |
| Three.js 交互复杂 | 中 | 影响建模体验 | 先固定草图平面，减少自由度 |
| WebSocket 断线重连 | 中 | 影响协同体验 | 加入连接状态和重新加载机制 |

### 8.2 项目管理风险

| 风险 | 等级 | 影响 | 应对 |
| --- | --- | --- | --- |
| 范围过大 | 高 | 无法按时完成 | 坚持 MVP 优先 |
| 成员分工不清 | 中 | 重复劳动或遗漏功能 | 每人负责明确模块 |
| 集成太晚 | 高 | 最后无法联调 | 每天合并一次 |
| 文档滞后 | 中 | 影响评分 | 文档与代码同步推进 |

### 8.3 部署风险

| 风险 | 等级 | 影响 | 应对 |
| --- | --- | --- | --- |
| 华为云配置不熟 | 中 | 部署延期 | 提前搭建 ECS |
| Nginx WebSocket 代理失败 | 中 | 协同无法公网演示 | 本地保留演示方案，提前验证代理 |
| 数据库丢失 | 中 | 演示失败 | 定期导出 SQL 或备份 JSON |
| 端口和安全组配置错误 | 中 | 外网无法访问 | 只开放 80/443/22，内部服务走 Docker 网络 |

## 9. 备选方案分析

### 9.1 前端框架

| 方案 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| Vue 3 + TypeScript | 学习曲线较平滑，单文件组件适合工作台界面 | CAD 复杂状态组织仍需设计 | 推荐 |
| React + TypeScript | 生态成熟，适合复杂组件 | 团队已选择 Vue 时切换成本不必要 | 可选 |
| 原生 JS | 简单直接 | 项目复杂后难维护 | 不推荐 |

### 9.2 后端框架

| 方案 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| Spring Boot | 企业级成熟，生态完整，适合权限、版本、协同服务 | Java 配置和工程结构需要规范 | 推荐 |
| NestJS | 模块化强，适合 TypeScript 全栈 | 团队已选择 Java 后端时不作为主线 | 可选 |
| Express | 简单快速 | 大项目结构容易散 | 不推荐作为主线 |
| Go Gin | 性能好 | 团队学习成本可能高 | 可选 |

### 9.3 协同方案

| 方案 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| WebSocket 操作广播 | 实现快，便于演示 | 冲突处理较弱 | MVP 推荐 |
| Yjs CRDT | 自动合并能力强 | 与 CAD 特征树适配需设计 | 增强项 |
| ShareDB OT | 适合 JSON 文档 | 规则设计复杂 | 后续可评估 |

### 9.4 数据存储方案

| 方案 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| PostgreSQL JSONB | 简单，结构化强 | 大模型不适合全存数据库 | MVP 推荐 |
| OBS 对象存储 | 适合大文件和快照 | 需要云 SDK 和权限配置 | 增强项 |
| 本地文件 | 开发简单 | 云部署可靠性一般 | 开发期可用 |

## 10. 最终可行性结论

本项目在课程实验范围内可行。

建议最终方案为：

1. 采用 Browser-Server 架构。
2. 前端使用 Vue 3 + TypeScript + Vite + Pinia + Three.js。
3. 后端使用 Spring Boot + Spring Security + Spring WebSocket + Spring Data JPA。
4. 数据库使用 PostgreSQL，迁移工具使用 Flyway。
5. 协同使用 STOMP over WebSocket 操作广播。
6. 模型数据使用特征树 JSON。
7. 部署使用华为云 ECS + Docker Compose + Nginx。
8. OBS、RDS、Redis、Yjs、OpenCascade.js/OCCT Wasm 作为增强项。

项目成功的判断标准不是“是否做成完整 CAD”，而是是否完成一条稳定的云协同 CAD 主链路：

创建项目 -> 绘制草图 -> 生成 3D 模型 -> 保存版本 -> 多人实时同步 -> 云端部署演示。
