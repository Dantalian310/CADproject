# 基于云的协同机械 CAD 系统技术选型文档

## 1. 文档说明

### 1.1 文档目的

本文档用于记录“基于云的协同机械 CAD 系统”的技术选型决策，说明每项核心技术的选择原因、替代方案、适用场景、风险和对后续编码的影响。

技术选型不是简单列工具，而是要回答：

1. 为什么选择这项技术。
2. 它在本项目中解决什么问题。
3. 与其他方案相比优劣如何。
4. 后续开发时应该如何使用它。
5. 它可能带来什么风险，如何规避。

### 1.2 项目约束

项目选型需要满足以下约束：

1. 课程项目周期有限，必须优先保证可交付。
2. 系统需要支持浏览器端 CAD 工作台。
3. 系统需要支持 3D 图形渲染。
4. 系统需要支持多人实时协同。
5. 系统需要支持云端保存和版本管理。
6. 团队倾向使用 Vue 前端和 Spring 后端。
7. 技术栈应便于解释、答辩和后续编码。

### 1.3 选型结论

最终选型：

| 层次 | 主选型 |
| --- | --- |
| 前端框架 | Vue 3 + TypeScript |
| 前端构建 | Vite |
| 前端路由 | Vue Router |
| 前端状态 | Pinia |
| UI 组件 | Element Plus + 自定义 CAD 样式 |
| 3D 渲染 | Three.js |
| HTTP 客户端 | Axios |
| 协同客户端 | @stomp/stompjs |
| 后端语言 | Java 21 |
| 后端框架 | Spring Boot |
| REST API | Spring Web MVC |
| 安全鉴权 | Spring Security + JWT |
| 实时协同 | Spring WebSocket + STOMP |
| ORM | Spring Data JPA + Hibernate |
| 数据库 | PostgreSQL |
| 数据库迁移 | Flyway |
| 缓存 | Redis，可选 |
| 可观测性 | Spring Boot Actuator + 结构化日志 |
| 部署 | 华为云 ECS + Nginx + Docker Compose |
| 对象存储 | PostgreSQL JSONB 起步，OBS 后续增强 |

## 2. 前端框架选型

### 2.1 选型结论

选择：

```text
Vue 3 + TypeScript
```

### 2.2 选择原因

1. Vue 3 上手快，适合课程团队快速搭建工作台界面。
2. 单文件组件适合组织工具栏、特征树、属性面板等 UI。
3. Composition API 适合把 CAD 交互逻辑抽成 composable。
4. TypeScript 可以约束复杂 CAD 数据结构和协同消息。
5. Vue 官方生态完整，Vue Router 和 Pinia 可直接配套使用。

### 2.3 备选方案对比

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Vue 3 + TypeScript | 上手快，官方生态清晰，适合复杂工作台 UI | CAD 状态设计仍需谨慎 | 采用 |
| React + TypeScript | 生态庞大，组件模型灵活 | 团队已决定 Vue，切换无必要 | 不采用 |
| 原生 JavaScript | 简单直接 | 状态复杂后难维护 | 不采用 |
| Angular | 工程化强 | 学习成本较高，课程周期不适合 | 不采用 |

### 2.4 编码影响

1. 前端页面以 `.vue` 单文件组件组织。
2. 复杂逻辑抽到 `cad/`、`api/`、`collaboration/` 等目录。
3. 组件不要直接操作后端返回的原始对象，应通过 Pinia action 修改状态。
4. CAD 模型类型必须用 TypeScript 明确定义。

### 2.5 主要风险

风险：

1. 把所有 CAD 状态都放在组件中，导致维护困难。
2. Vue 响应式系统和 Three.js 对象生命周期混杂。

规避：

1. Pinia 管理业务状态。
2. Three.js 对象由 SceneManager 管理。
3. Vue 组件只负责挂载和交互入口。

## 3. 构建工具选型

### 3.1 选型结论

选择：

```text
Vite
```

### 3.2 选择原因

1. Vite 对 Vue 3 支持成熟。
2. 启动和热更新快。
3. 配置简单。
4. 生产构建输出静态文件，适合 Nginx 部署。

### 3.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Vite | 快速、轻量、适合 Vue 3 | 需要了解环境变量和代理配置 | 采用 |
| Webpack | 成熟强大 | 配置复杂 | 不采用 |
| Vue CLI | 传统 Vue 工具 | 已不是新项目首选 | 不采用 |

### 3.4 编码影响

1. 前端环境变量必须以 `VITE_` 开头。
2. 开发期通过 Vite proxy 转发 `/api` 和 `/ws`。
3. 生产部署时由 Nginx 处理静态资源和代理。

## 4. 前端状态管理选型

### 4.1 选型结论

选择：

```text
Pinia
```

### 4.2 选择原因

1. Pinia 是 Vue 3 项目常用状态管理方案。
2. API 简洁，TypeScript 体验较好。
3. 适合拆分 auth、project、cad、collaboration 多个 store。
4. 比 Vuex 更轻量，适合本项目规模。

### 4.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Pinia | Vue 3 生态主流，简洁 | 需要设计好 store 边界 | 采用 |
| Vuex | 传统成熟 | Vue 3 新项目不优先 | 不采用 |
| 组件局部状态 | 简单 | CAD 状态跨组件共享困难 | 不采用 |

### 4.4 编码影响

建议 store：

1. `auth.store.ts`
2. `project.store.ts`
3. `cad.store.ts`
4. `collaboration.store.ts`

所有模型变更必须通过 `cad.store` 的 action。

## 5. UI 组件库选型

### 5.1 选型结论

选择：

```text
Element Plus + 自定义 CAD 工作台样式
```

### 5.2 选择原因

1. Element Plus 适合快速完成登录表单、项目表格、弹窗、消息提示。
2. Vue 3 兼容性好。
3. 可以减少通用管理界面开发时间。
4. CAD 工作台核心区域仍采用自定义布局，避免后台风格过重。

### 5.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Element Plus | Vue 生态成熟，组件齐全 | 默认风格偏后台 | 采用 |
| Naive UI | 设计现代 | 团队熟悉度可能较低 | 可选 |
| Ant Design Vue | 企业后台能力强 | 视觉和依赖较重 | 不优先 |
| 完全自写 | 风格可控 | 时间成本高 | 不采用 |

### 5.4 编码影响

1. 登录、项目、版本弹窗可以用 Element Plus。
2. CAD 工具栏、特征树、属性面板建议自定义样式。
3. 不要用大型卡片式布局堆工作台，CAD 工具应紧凑。

## 6. 3D 渲染技术选型

### 6.1 选型结论

选择：

```text
Three.js
```

### 6.2 选择原因

1. Three.js 是 Web 3D 渲染成熟方案。
2. 能快速实现 scene、camera、renderer、mesh、material。
3. 支持 OrbitControls、Raycaster、ExtrudeGeometry。
4. 配合轻量 CSG 库可以实现课程级切除、布尔并集和差集。
5. 足以完成课程项目的轻量 CAD 原型。
6. 比直接使用 WebGL 成本低得多。

### 6.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Three.js | 轻量、成熟、示例丰富 | 不是工业 CAD 内核 | 采用 |
| three-bvh-csg | 可在 Three.js mesh 上做并、差、交等 CSG 操作 | 实验性质，要求几何体封闭且可能有数值边界问题 | MVP 可采用 |
| Babylon.js | 引擎能力强 | 对本项目略重 | 不采用 |
| 原生 WebGL | 底层可控 | 开发成本高 | 不采用 |
| OpenCascade.js / OCCT Wasm | 工业几何能力强，已有 WebAssembly 封装和示例 | Wasm 体积、bundler 配置和 API 学习成本较高 | 后续增强 |

### 6.4 编码影响

1. 首版用 Three.js 生成长方体、圆柱体和简单拉伸体。
2. 需要封装 `SceneManager`，避免 Vue 组件直接管理复杂 Three.js 对象。
3. 需要定义 `GeometryKernel` 接口，为后续 OCCT Wasm 预留扩展。
4. Cut 可以作为差集布尔的特化操作实现。
5. 布尔并集和差集首版只支持基础封闭实体。

### 6.5 主要风险

1. Three.js Mesh 不等于真实 CAD B-Rep。
2. mesh 级 CSG 不等价于工业级 B-Rep，可能存在数值和非流形问题。
3. 倒角、拓扑编辑不是 Three.js 强项。
4. 需要在报告中明确“首版是轻量参数化原型”。

## 7. 后端语言与框架选型

### 7.1 选型结论

选择：

```text
Java 21 + Spring Boot
```

### 7.2 选择原因

1. 团队倾向 Java 后端。
2. Spring Boot 工程化成熟，适合项目管理、权限、版本、协同服务。
3. Spring 生态中安全、数据库、WebSocket、缓存能力完整。
4. Java 强类型适合后端业务建模。
5. 适合在答辩中体现企业级架构设计能力。

### 7.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Spring Boot | 企业级成熟，生态完整 | 工程结构较重 | 采用 |
| NestJS | TypeScript 全栈一致 | 团队已选择 Java 后端 | 不采用 |
| Express | 快速简单 | 大项目结构易散 | 不采用 |
| Go Gin | 性能好 | 团队学习成本不确定 | 不采用 |

### 7.4 编码影响

1. 后端按模块分包。
2. Controller、Service、Repository 严格分层。
3. DTO 和 Entity 分离。
4. 使用 Maven 管理依赖。
5. 使用 `application.yml` 管理配置。

## 8. REST API 技术选型

### 8.1 选型结论

选择：

```text
Spring Web MVC
```

### 8.2 选择原因

1. 与 Spring Boot 深度集成。
2. 适合请求响应式业务接口。
3. 注解式 Controller 开发清晰。
4. 与 Spring Security、Validation、JPA 配合成熟。

### 8.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Spring Web MVC | 成熟稳定，适合 REST API | 阻塞模型，但本项目足够 | 采用 |
| Spring WebFlux | 高并发响应式 | 学习成本和复杂度较高 | 不采用 |
| GraphQL | 查询灵活 | 对本项目非必要 | 不采用 |

### 8.4 编码影响

1. HTTP 接口以 `/api` 为前缀。
2. 请求对象使用 DTO + Validation。
3. 返回统一响应结构。
4. 使用全局异常处理。

## 9. 安全鉴权选型

### 9.1 选型结论

选择：

```text
Spring Security + JWT
```

### 9.2 选择原因

1. Spring Security 是 Spring 项目事实标准安全框架。
2. JWT 适合前后端分离项目。
3. token 可用于 HTTP API，也可用于 WebSocket 握手鉴权。
4. 不需要服务端保存传统 session。

### 9.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Spring Security + JWT | 成熟、无状态、适合前后端分离 | 配置需要理解 | 采用 |
| Session 登录 | 简单 | 前后端分离和 WebSocket 处理不如 JWT 清晰 | 不采用 |
| OAuth2/OIDC | 标准完整 | 对课程项目过重 | 不采用 |

### 9.4 编码影响

1. 登录成功返回 JWT。
2. 前端 Axios 拦截器附加 `Authorization: Bearer <token>`。
3. 后端用 Filter 校验 token。
4. WebSocket 连接需要校验 token。
5. 项目和文档接口必须校验权限。

## 10. 实时协同技术选型

### 10.1 选型结论

选择：

```text
Spring WebSocket + STOMP
```

前端使用：

```text
@stomp/stompjs
```

### 10.2 选择原因

1. WebSocket 适合低延迟双向通信。
2. STOMP 提供 destination、subscribe、send 语义，适合文档房间模型。
3. Spring 原生支持 STOMP over WebSocket。
4. 比原生 WebSocket 更便于讲清协同架构。

### 10.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Spring WebSocket + STOMP | 与 Spring 适配好，语义清晰 | 前端需理解 STOMP | 采用 |
| 原生 WebSocket | 简单直接 | 房间和消息路由需自研 | 不优先 |
| Socket.IO | 前端生态常见 | Java 后端原生支持弱 | 不采用 |
| Yjs WebSocket | CRDT 能力强 | CAD 特征树适配复杂 | 后续增强 |

### 10.4 编码影响

1. 文档协同以 documentId 划分 topic。
2. 前端订阅 `/topic/documents/{documentId}/operations`。
3. 前端发送到 `/app/documents/{documentId}/operations`。
4. 服务端先做操作广播和日志记录。
5. MVP 不做完整 CRDT，仅做操作事件同步和简单冲突提示。

## 11. 数据访问技术选型

### 11.1 选型结论

选择：

```text
Spring Data JPA + Hibernate
```

### 11.2 选择原因

1. Spring 官方数据访问生态成熟。
2. Repository 接口能减少 CRUD 样板代码。
3. 适合用户、项目、文档、版本等实体建模。
4. 与事务管理集成自然。

### 11.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Spring Data JPA | 官方生态，实体建模清晰 | 复杂 SQL 控制不如 MyBatis 直接 | 采用 |
| MyBatis-Plus | CRUD 快，SQL 可控，国内常用 | 领域模型表达不如 JPA 自然 | 可选 |
| JDBC Template | 轻量直接 | 样板代码较多 | 不采用 |

### 11.4 编码影响

1. 使用 Entity 映射表。
2. 使用 Repository 访问数据库。
3. 使用 Service 管理事务。
4. Entity 不直接返回前端。
5. 对 JSONB 字段需要单独验证映射策略。

## 12. 数据库选型

### 12.1 选型结论

选择：

```text
PostgreSQL
```

### 12.2 选择原因

1. 同时适合关系数据和 JSONB 模型快照。
2. 用户、项目、权限、版本适合关系模型。
3. CAD 文档快照适合 JSONB。
4. 事务能力强。
5. 云部署和 Docker 部署都方便。

### 12.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| PostgreSQL | JSONB 强，关系能力强 | 团队需熟悉部分 PG 特性 | 采用 |
| MySQL | 普及度高 | JSON 能力和 PG 相比略弱 | 不优先 |
| MongoDB | 文档模型灵活 | 权限、版本等关系数据不如 PG 自然 | 不采用 |
| SQLite | 简单 | 不适合云端多人系统 | 不采用 |

### 12.4 编码影响

1. 使用 Flyway 创建表结构。
2. 模型快照使用 JSONB。
3. 版本保存使用事务。
4. documentId、projectId、versionNumber 需要索引。

## 13. 数据库迁移工具选型

### 13.1 选型结论

选择：

```text
Flyway
```

### 13.2 选择原因

1. Spring Boot 集成简单。
2. SQL 迁移文件清晰。
3. 团队成员数据库结构可以保持一致。
4. 部署到云端时可自动迁移。

### 13.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Flyway | 简洁，SQL 明确 | 已执行脚本不能随意改 | 采用 |
| Liquibase | 功能更强 | 配置更复杂 | 不采用 |
| Hibernate ddl-auto | 快速 | 不适合作为正式结构管理 | 开发期也不建议依赖 |

### 13.4 编码影响

1. 初始脚本命名为 `V1__init_schema.sql`。
2. 后续变更新增 `V2__xxx.sql`。
3. 不随意修改已执行的迁移文件。

## 14. 缓存与协同状态选型

### 14.1 选型结论

选择：

```text
Redis，可选
```

### 14.2 选择原因

1. MVP 单实例可以先用内存维护在线用户。
2. Redis 后续可用于在线用户、房间状态、短期锁。
3. Redis 支持过期时间，适合临时状态。

### 14.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| 后端内存 | 实现快 | 多实例不共享 | MVP 采用 |
| Redis | 快，适合短期状态 | 多一个部署组件 | P1/P2 增强 |
| 数据库表 | 持久 | 不适合高频在线状态 | 不采用 |

### 14.4 编码影响

1. Presence 服务先抽象接口。
2. MVP 实现 `InMemoryPresenceService`。
3. 后续替换为 `RedisPresenceService`。

## 15. 文件存储选型

### 15.1 选型结论

选择：

```text
MVP：PostgreSQL JSONB
增强：华为云 OBS
```

### 15.2 选择原因

1. MVP 模型数据较小，直接存 JSONB 最简单。
2. 保存、加载、版本查询都在一个数据库事务中完成。
3. 后续大模型、导出文件、附件适合 OBS。

### 15.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| PostgreSQL JSONB | 简单，事务一致 | 大文件不适合 | MVP 采用 |
| OBS | 适合大文件，云原生 | 接入和权限配置更复杂 | 后续增强 |
| ECS 本地文件 | 简单 | 可靠性和迁移性较差 | 开发期可用 |

### 15.4 编码影响

1. `DocumentVersion` 保留 `snapshotJson` 和 `snapshotPath`。
2. 定义 `StorageService` 抽象。
3. 先实现数据库 JSONB 保存。
4. 后续接入 OBS 时不影响业务层。

## 16. 部署技术选型

### 16.1 选型结论

选择：

```text
华为云 ECS + Nginx + Docker Compose
```

### 16.2 选择原因

1. 贴合课程要求中的华为云。
2. ECS 部署方式直观，适合演示。
3. Nginx 可统一代理前端、API、WebSocket。
4. Docker Compose 比 Kubernetes 更适合课程周期。

### 16.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| ECS + Nginx + Docker Compose | 简单、可控、适合演示 | 单机扩展能力有限 | 采用 |
| CCE/Kubernetes | 云原生、扩展强 | 学习和配置成本高 | 后续增强 |
| 纯本地演示 | 成本低 | 不满足云端特色 | 仅备用 |
| Serverless | 运维少 | WebSocket 和文件部署不如 ECS 直观 | 不采用 |

### 16.4 编码影响

1. 前端构建成静态文件。
2. 后端打成 Jar 或 Docker 镜像。
3. 数据库用 Docker volume 持久化。
4. Nginx 配置 WebSocket Upgrade。
5. 环境变量不能写死。

## 17. 几何内核选型

### 17.1 选型结论

选择：

```text
MVP：Three.js 参数化几何 + 轻量 CSG
增强：OpenCascade.js / OCCT Wasm
```

### 17.2 选择原因

1. OpenCascade.js 已经提供 OCCT 的 JavaScript/WebAssembly 移植版本，比从零编译 OCCT 更具体可行，但仍不适合阻塞首版。
2. MVP 要求简单草图、拉伸、切除、布尔并集和差集。
3. Three.js 足以演示矩形/圆拉伸。
4. 轻量 CSG 库可支撑基础封闭体的并集和差集演示。
5. OpenCascade.js 官方示例展示了在浏览器中创建 shape、执行 Boolean Cut，并导出 GLB 可视化的路径。
6. 架构上预留 GeometryKernel 接口，未来可替换。

### 17.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| Three.js 参数化几何 | 快速可演示 | 非工业级 CAD 内核 | MVP 采用 |
| three-bvh-csg 等轻量 CSG 库 | 可做基础 Cut、Union、Subtract | 稳定性和拓扑能力有限 | MVP 可采用 |
| OpenCascade.js | 工业级 B-Rep 能力，WebAssembly 封装更直接 | Wasm 体积、初始化、Vite 配置和 API 学习成本高 | P1/P2 技术验证 |
| 自行编译 OCCT Wasm | 可完全定制 | 编译链复杂，时间成本高 | 不作为近期方案 |
| 自研 B-Rep | 理论可控 | 远超课程项目范围 | 不采用 |

### 17.4 编码影响

1. 定义 `GeometryKernel` 接口。
2. `ThreeGeometryKernel` 作为首版实现。
3. 不把 Three.js 几何生成逻辑散落到组件中。
4. 在 `GeometryKernel` 中提供 `buildCut` 和 `buildBoolean`。
5. 新增 `OpenCascadeJsGeometryKernel` 作为后续验证目标。
6. 后续 OCCT 接入时替换实现，不改上层 CAD 状态。

### 17.5 OpenCascade.js 技术验证边界

验证目标：

1. 安装 `opencascade.js@beta`。
2. 在独立 demo 中完成 Box - Sphere Cut。
3. 将结果导出为 GLB 或转换为 Three.js 可渲染 mesh。
4. 评估 Wasm 加载时间、包体积、内存占用和 Vite 配置成本。
5. 验证 `.wasm` 文件的构建输出、缓存策略、加载进度提示和失败回退。

是否进入主线的判断标准：

1. 不影响 MVP 进度。
2. 能稳定在浏览器端加载。
3. 能把结果接入现有 `GeometryKernel`。
4. 能用文档清楚解释其与 Three.js mesh/CSG 的区别。
5. 能提供明确的 `ThreeGeometryKernel` 回退方案。

## 18. 可观测性与工业软件运行状态选型

### 18.1 选型结论

选择：

```text
Spring Boot Actuator + 结构化日志 + OperationLog
```

### 18.2 选择原因

1. Rapid SCADA 体现了工业软件对 Web 访问、实时状态、日志和可运维性的重视。
2. 本项目虽然是 CAD，但协同编辑同样需要可观测性。
3. Actuator 可用于健康检查，便于云端部署验证。
4. OperationLog 可用于协同调试、答辩演示和问题追踪。

### 18.3 编码影响

1. 后端保留 `/actuator/health`。
2. 关键操作记录到 `operation_logs`。
3. WebSocket 连接、断开、冲突提示应写入结构化日志。
4. 前端状态栏显示 API 连接、WebSocket 连接、保存状态。

## 19. 协同算法选型

### 19.1 选型结论

选择：

```text
MVP：操作事件广播 + 简单冲突提示
增强：Yjs/CRDT
```

### 19.2 选择原因

1. CAD 特征树协同比文本协同复杂。
2. MVP 目标是两人实时同步，不是完整无冲突合并。
3. 操作事件广播实现快，便于演示和调试。
4. 后续可逐步引入 Yjs 管理共享文档。

### 19.3 备选方案

| 方案 | 优点 | 缺点 | 决策 |
| --- | --- | --- | --- |
| 操作广播 | 简单、可控、易演示 | 冲突处理弱 | MVP 采用 |
| 对象级锁 | 避免同对象并发修改 | 协同体验略保守 | P1 可加 |
| Yjs CRDT | 合并能力强 | CAD 特征树适配复杂 | 后续增强 |
| OT | 理论成熟 | 操作转换规则复杂 | 暂不采用 |

### 19.4 编码影响

1. 所有本地 CAD 操作都生成 operation。
2. operation 包含 `documentId`、`userId`、`baseVersion`、`targetId`、`payload`。
3. 服务端维护 document room 的连接映射，并广播增量 operation。
4. 服务端记录操作日志。
5. 客户端应用远程 operation。
6. 光标消息节流后广播。
7. 同一对象并发修改时提示冲突。

## 20. 选型风险汇总

| 风险 | 来源 | 应对 |
| --- | --- | --- |
| Vue 状态复杂 | CAD 文档模型复杂 | Pinia 分模块，操作统一 action |
| Three.js 不是 CAD 内核 | 渲染库能力边界 | 明确 MVP 范围，预留 GeometryKernel |
| Spring Security 配置复杂 | 鉴权链路 | 先实现最小 JWT，再扩展权限 |
| STOMP 学习成本 | 协同协议 | 先跑通 join/send/subscribe 最小闭环 |
| JPA JSONB 映射 | 模型快照 | 先用字符串或 JsonNode 保存，后续优化 |
| 云部署出错 | Nginx/WebSocket/安全组 | 本地 Docker 先跑通，再部署 ECS |

## 21. 技术选型结论

本项目选择 Vue 3 和 Spring Boot 是合理的：

1. Vue 3 能快速搭建 CAD 工作台。
2. Three.js 能完成浏览器端图形渲染。
3. Spring Boot 能稳定承载用户、项目、权限、版本和协同业务。
4. PostgreSQL 同时满足关系数据和模型 JSON 快照存储。
5. STOMP over WebSocket 适合文档房间式协同。
6. Docker Compose + ECS 能在课程周期内完成云端部署。

开发时应坚持：

1. 先实现 MVP 主链路。
2. 不让 OpenCascade.js/OCCT、CRDT、OBS、Kubernetes 阻塞首版。
3. 每项技术都服务于具体功能，不为了技术堆叠而引入复杂组件。
