# 基于云的协同机械 CAD 系统技术栈说明文档

## 1. 文档说明

本文档用于说明本项目采用的主要技术栈，包括每项技术在项目中的作用、相对其他技术的优势、核心技术细节和主要难点。

当前项目技术路线：

```text
前端：Vue 3 + TypeScript + Vite + Vue Router + Pinia + Three.js + three-bvh-csg + Element Plus
协同客户端：@stomp/stompjs
后端：Java 21 + Spring Boot + Spring Web MVC + Spring Security + Spring WebSocket
持久层：Spring Data JPA + Hibernate + PostgreSQL + Flyway
缓存：Redis，可选
部署：华为云 ECS + Nginx + Docker Compose
存储：PostgreSQL JSONB，后续可扩展到华为云 OBS
```

## 2. Vue 3

### 2.1 在项目中的作用

Vue 3 用于构建浏览器端 CAD 工作台界面，包括：

1. 登录页。
2. 项目列表页。
3. CAD 工作台主页面。
4. 顶部工具栏。
5. 左侧特征树。
6. 中央视图区容器。
7. 右侧属性面板。
8. 底部状态栏。
9. 在线用户与协同状态显示。

### 2.2 相比其他技术的优势

相比 React：

1. Vue 单文件组件将模板、逻辑和样式放在同一个 `.vue` 文件中，适合课程团队快速协作。
2. Vue 的模板语法更接近 HTML，上手成本较低。
3. Vue 官方生态包含 Vue Router、Pinia 等配套方案，组合清晰。

相比原生 JavaScript：

1. Vue 的响应式系统可以自动驱动界面更新。
2. 组件化可以降低 CAD 工作台复杂 UI 的维护成本。
3. 状态变化和视图渲染关系更清楚。

### 2.3 核心技术细节

需要重点掌握：

1. 单文件组件 SFC。
2. Composition API。
3. `ref`、`reactive`、`computed`、`watch`。
4. 组件 props 和 emits。
5. 生命周期钩子。
6. 与 Three.js 这种命令式渲染库的集成方式。

### 2.4 主要难点

1. CAD 状态复杂，不能把所有状态都塞进一个组件。
2. Three.js 场景不应完全交给 Vue 响应式系统管理，否则容易出现性能和生命周期问题。
3. 需要明确哪些状态属于 Vue UI，哪些状态属于 Three.js 渲染层，哪些状态属于 CAD 文档模型。

建议：

1. Vue 负责界面和交互入口。
2. Pinia 负责核心 CAD 状态。
3. Three.js 负责渲染场景。
4. CAD 数据模型保持纯 TypeScript 对象，便于保存、协同和测试。

## 3. TypeScript

### 3.1 在项目中的作用

TypeScript 用于前端类型约束，尤其是：

1. 草图实体类型。
2. 特征树节点类型。
3. 版本快照类型。
4. 协同消息类型。
5. API 请求和响应类型。

### 3.2 优势

相比 JavaScript：

1. 可以提前发现字段名错误。
2. 可以让 CAD 数据模型更清晰。
3. 协同消息结构更容易维护。
4. 重构成本更低。

### 3.3 核心技术细节

需要定义稳定的类型，例如：

```ts
export type SketchEntity =
  | RectangleEntity
  | CircleEntity
  | LineEntity

export interface ExtrudeFeature {
  id: string
  type: 'extrude'
  sourceSketchId: string
  sourceEntityId: string
  depth: number
}
```

### 3.4 主要难点

1. CAD 模型会不断扩展，类型设计不能过早僵死。
2. 前端类型和后端 DTO 需要保持一致。
3. 协同消息 payload 是多态结构，需要设计 discriminated union。

## 4. Vite

### 4.1 在项目中的作用

Vite 用于创建和运行 Vue 前端项目。

主要作用：

1. 本地开发服务器。
2. 热更新。
3. TypeScript 构建。
4. 生产环境静态资源打包。

### 4.2 优势

相比传统 Webpack 配置：

1. 启动快。
2. 热更新快。
3. Vue 模板支持成熟。
4. 配置较轻。

### 4.3 核心技术细节

需要掌握：

1. `vite.config.ts`。
2. 环境变量 `VITE_` 前缀。
3. 开发代理。
4. 生产构建。

开发期可以配置代理：

```ts
server: {
  proxy: {
    '/api': 'http://localhost:8080',
    '/ws': {
      target: 'http://localhost:8080',
      ws: true
    }
  }
}
```

### 4.4 主要难点

1. 前端路由 history 模式部署到 Nginx 后需要配置 fallback。
2. WebSocket 代理需要显式开启 `ws: true`。
3. 生产环境 API 地址不能写死。

## 5. Vue Router

### 5.1 在项目中的作用

Vue Router 管理页面路由：

1. `/login`：登录页。
2. `/projects`：项目列表页。
3. `/workspace/:documentId`：CAD 工作台页。

### 5.2 优势

相比手动切换组件：

1. URL 和页面状态一致。
2. 可以刷新页面后恢复当前工作台。
3. 可以做登录守卫。

### 5.3 核心技术细节

1. `createRouter`。
2. `createWebHistory`。
3. 路由守卫。
4. 动态路由参数。

### 5.4 主要难点

1. 未登录用户进入工作台时需要跳转登录页。
2. 打开工作台前需要加载项目权限和文档数据。
3. Nginx 部署时需要把未知路径回退到 `index.html`。

## 6. Pinia

### 6.1 在项目中的作用

Pinia 用于前端状态管理。

建议拆分：

1. `auth.store.ts`：登录用户、token。
2. `project.store.ts`：项目列表、当前项目。
3. `cad.store.ts`：草图、特征、选中对象、Undo/Redo。
4. `collaboration.store.ts`：在线用户、连接状态、远程光标。

### 6.2 优势

相比把状态散落在组件中：

1. 状态集中，便于调试。
2. 跨组件共享方便。
3. 与 Vue 3 Composition API 配合自然。

相比 Vuex：

1. API 更简洁。
2. TypeScript 体验更好。
3. 适合作为 Vue 3 项目的默认状态管理方案。

### 6.3 核心技术细节

1. `defineStore`。
2. state、getters、actions。
3. store 持久化，可选。
4. 模块化拆分。

### 6.4 主要难点

1. CAD 状态变化频繁，需要避免无意义的全局重渲染。
2. Undo/Redo 栈和协同消息都依赖状态变更，必须统一操作入口。
3. 保存模型时需要从 store 序列化出稳定 JSON。

## 7. Three.js

### 7.1 在项目中的作用

Three.js 是前端 3D 渲染核心。

用于实现：

1. 3D 场景。
2. 相机。
3. 光照。
4. 网格辅助线。
5. 坐标轴。
6. 草图实体显示。
7. 拉伸模型显示。
8. 对象选中和高亮。

### 7.2 优势

相比直接使用 WebGL：

1. API 更友好。
2. 有成熟的几何体、材质、相机、控制器。
3. 示例和社区丰富。

相比 Babylon.js：

1. Three.js 更轻量。
2. CAD 原型需要的是可控渲染，不需要完整游戏引擎能力。
3. 课程资料中也推荐 Three.js。

### 7.3 核心技术细节

需要掌握：

1. `Scene`。
2. `PerspectiveCamera` 或 `OrthographicCamera`。
3. `WebGLRenderer`。
4. `Mesh`、`Geometry`、`Material`。
5. `OrbitControls`。
6. `Raycaster`。
7. `Shape` 和 `ExtrudeGeometry`。
8. 坐标转换。

### 7.4 主要难点

1. 鼠标屏幕坐标转换为草图平面坐标。
2. 选中对象需要 Raycaster 或自定义命中测试。
3. Three.js 对象生命周期需要手动释放，避免内存泄漏。
4. CAD 参数变化后需要重新生成几何体。
5. Three.js Mesh 不是工业级 B-Rep，后续接入 OpenCASCADE 需要抽象几何层。

## 8. three-bvh-csg

### 8.1 在项目中的作用

three-bvh-csg 用于补足 Three.js 在简单布尔建模方面的能力。

本项目中用于：

1. 布尔并集 Union。
2. 布尔差集 Subtract。
3. Cut 切除，作为差集操作的特化。
4. 课程级基础封闭体布尔演示。

### 8.2 优势

相比完全自写 CSG：

1. 实现成本低。
2. 与 Three.js mesh 体系兼容。
3. 能较快支撑实验 PDF 要求的并集和差集。

相比 OpenCascade.js/OCCT Wasm：

1. 接入难度低。
2. 不需要 C++/Wasm 编译链。
3. 更适合课程项目 MVP。

### 8.3 主要技术细节

需要关注：

1. 输入几何体应尽量是封闭、水密 mesh。
2. 布尔前需要更新 mesh 世界矩阵。
3. 布尔结果仍是 Three.js mesh，不是工业 B-Rep。
4. 需要把布尔操作记录成 Feature，而不是只保存结果 mesh。

### 8.4 主要难点

1. mesh 级布尔存在数值精度和边界问题。
2. 非封闭几何体可能导致结果异常。
3. 布尔结果不适合解释为工业级 CAD 内核能力。
4. 后续接入 OpenCASCADE 时，需要替换 GeometryKernel 实现。

## 9. Element Plus

### 9.1 在项目中的作用

Element Plus 用于通用 UI：

1. 登录表单。
2. 项目列表。
3. 创建项目弹窗。
4. 属性面板输入框。
5. 版本历史表格。
6. 消息提示。

CAD 工作台的核心布局和工具栏建议自定义，不完全依赖 Element Plus。

### 9.2 优势

相比完全自写 UI：

1. 表单、弹窗、表格能快速完成。
2. Vue 生态适配好。
3. 样式稳定，适合后台和管理类页面。

### 9.3 主要难点

1. Element Plus 默认视觉偏管理后台，CAD 工作台需要更紧凑的专业工具界面。
2. 需要控制组件尺寸，避免工具栏和面板过大。
3. 需要与自定义 Three.js 视图区布局协调。

## 10. Axios

### 10.1 在项目中的作用

Axios 用于调用后端 REST API。

主要接口：

1. 注册。
2. 登录。
3. 获取当前用户。
4. 项目 CRUD。
5. 文档保存和加载。
6. 版本查询和恢复。

### 10.2 优势

相比原生 `fetch`：

1. 拦截器使用方便。
2. 统一添加 JWT token 更简单。
3. 错误处理更清晰。

### 10.3 主要难点

1. token 过期处理。
2. 统一错误提示。
3. API base URL 需要按环境切换。

## 11. STOMP Client

### 11.1 在项目中的作用

前端使用 `@stomp/stompjs` 连接 Spring WebSocket。

用于：

1. 加入文档协同房间。
2. 订阅模型操作广播。
3. 订阅在线用户列表。
4. 发送草图、特征、光标操作。

### 11.2 优势

相比原生 WebSocket：

1. 有 topic 和 destination 概念。
2. 与 Spring WebSocket 天然适配。
3. 消息语义更清晰。

相比 Socket.IO：

1. Spring 原生支持 STOMP，而不是 Socket.IO。
2. 不需要额外 Java Socket.IO 服务端实现。

### 11.3 主要难点

1. 连接时携带 JWT。
2. 断线重连。
3. 订阅取消。
4. 防止重复订阅。
5. 协同消息顺序和版本冲突。

## 12. Java 21

### 12.1 在项目中的作用

Java 21 作为后端运行语言。

### 12.2 优势

1. LTS 版本，适合课程和长期维护。
2. Spring 生态支持成熟。
3. 强类型适合复杂业务建模。
4. 团队 Java 基础更好时开发效率更稳定。

### 12.3 主要难点

1. Java 项目结构比 Node 项目更重。
2. Spring Security 和 JPA 配置需要理解。
3. DTO、Entity、Repository、Service 分层需要规范。

## 13. Spring Boot

### 13.1 在项目中的作用

Spring Boot 是后端主框架。

负责：

1. 启动后端服务。
2. 管理依赖和自动配置。
3. 承载 REST API。
4. 集成安全、数据库、WebSocket、缓存。
5. 提供健康检查和配置管理。

### 13.2 优势

相比传统 Spring：

1. 自动配置减少样板配置。
2. 内嵌 Tomcat，直接运行 Jar。
3. Starter 依赖简化工程搭建。

相比 Express/NestJS：

1. Java 企业级生态成熟。
2. Spring Security、Spring Data、Spring WebSocket 一体化程度高。
3. 更适合做权限、版本、项目管理等后端业务。

### 13.3 核心技术细节

1. Controller。
2. Service。
3. Repository。
4. Configuration。
5. Dependency Injection。
6. `application.yml`。
7. Profiles。

### 13.4 主要难点

1. 分层过度会让小项目显得复杂，需要把握粒度。
2. 自动配置背后的机制需要理解，遇到问题不能只看表面报错。
3. 后端实体、DTO 和前端模型需要保持同步。

## 14. Spring Web MVC

### 14.1 在项目中的作用

Spring Web MVC 用于 REST API：

1. `/api/auth/register`
2. `/api/auth/login`
3. `/api/projects`
4. `/api/documents/{id}`
5. `/api/documents/{id}/save`
6. `/api/documents/{id}/versions`

### 14.2 优势

1. 注解式开发简单。
2. 与 Spring Security、Validation、JPA 集成成熟。
3. 适合传统请求响应式后端。

### 14.3 主要难点

1. DTO 校验。
2. 统一异常处理。
3. 跨域配置。
4. 前后端字段命名一致性。

## 15. Spring Security + JWT

### 15.1 在项目中的作用

用于：

1. 用户登录。
2. 密码加密。
3. JWT 生成和校验。
4. 接口权限控制。
5. WebSocket 连接鉴权。

### 15.2 优势

相比手写鉴权：

1. 安全模型成熟。
2. 支持过滤器链。
3. 可扩展认证和授权逻辑。
4. 与 Spring MVC 集成好。

### 15.3 核心技术细节

1. `SecurityFilterChain`。
2. `PasswordEncoder`。
3. JWT filter。
4. `Authentication`。
5. 角色和权限。
6. CORS 和 CSRF 配置。

### 15.4 主要难点

1. 前后端分离项目通常需要关闭或调整 CSRF。
2. WebSocket 握手鉴权和 HTTP 鉴权不是完全一样。
3. token 过期后前端需要统一处理。
4. 权限模型不能做得过重，否则影响进度。

## 16. Spring WebSocket + STOMP

### 16.1 在项目中的作用

用于多人协同设计。

典型消息：

1. 用户加入文档。
2. 用户离开文档。
3. 用户移动光标。
4. 用户新增草图。
5. 用户修改特征。
6. 用户保存版本。

### 16.2 优势

相比轮询：

1. 延迟低。
2. 双向通信。
3. 更适合实时协同。

相比原生 WebSocket：

1. STOMP 有 destination 和订阅语义。
2. 可以按文档 ID 建 topic。
3. Spring 支持成熟。

### 16.3 核心技术细节

1. WebSocket endpoint。
2. Message broker。
3. `@MessageMapping`。
4. `SimpMessagingTemplate`。
5. Topic 命名。
6. Presence 在线用户。

### 16.4 主要难点

1. 消息顺序和重复消息处理。
2. 同一对象并发修改的冲突提示。
3. 断线重连后状态恢复。
4. 多实例部署时需要 Redis 或外部消息系统协助。

## 17. Spring Data JPA + Hibernate

### 17.1 在项目中的作用

用于数据库访问。

实体包括：

1. User。
2. Project。
3. ProjectMember。
4. Document。
5. DocumentVersion。
6. OperationLog。

### 17.2 优势

相比手写 JDBC：

1. 减少大量 CRUD 样板代码。
2. Repository 模型清晰。
3. 事务管理成熟。
4. 与 Spring Boot 集成好。

相比 MyBatis-Plus：

1. 更贴合 Spring 官方生态。
2. 面向领域实体建模更自然。
3. 对课程文档中的“实体关系”和“版本快照”解释更方便。

MyBatis-Plus 仍可作为备选，尤其适合团队更熟悉 SQL 和国内 Java 开发生态的情况。

### 17.3 核心技术细节

1. `@Entity`。
2. `@Id`。
3. `JpaRepository`。
4. `@Transactional`。
5. 实体关系。
6. JSONB 字段映射。

### 17.4 主要难点

1. 不能滥用双向关联，避免序列化循环。
2. 复杂查询需要谨慎设计。
3. PostgreSQL JSONB 和 Java 对象映射需要测试。
4. Entity 不应直接暴露给前端，应该用 DTO。

## 18. PostgreSQL

### 18.1 在项目中的作用

PostgreSQL 是主数据库。

保存：

1. 用户。
2. 项目。
3. 成员权限。
4. 文档元数据。
5. 版本快照。
6. 操作日志。

### 18.2 优势

相比 MySQL：

1. JSONB 支持强，适合保存 CAD 模型快照。
2. 事务能力成熟。
3. 数据类型丰富。
4. 开源生态稳定。

相比 MongoDB：

1. 既能保存关系数据，也能保存 JSON 快照。
2. 用户、项目、权限、版本这些关系型数据更自然。

### 18.3 核心技术细节

1. 表设计。
2. 主键和外键。
3. JSONB 字段。
4. 索引。
5. 事务。
6. 版本号约束。

### 18.4 主要难点

1. 模型 JSON 结构需要稳定。
2. JSONB 查询能力强，但不要把所有业务都塞进 JSON。
3. 保存版本时要保证事务一致性。

## 19. Flyway

### 19.1 在项目中的作用

Flyway 用于数据库迁移。

它管理：

1. 建表脚本。
2. 字段变更。
3. 索引创建。
4. 初始数据。

### 19.2 优势

相比手动改数据库：

1. 所有人数据库结构一致。
2. 云端部署可自动迁移。
3. 便于答辩说明工程规范。

相比 Hibernate 自动建表：

1. 更可控。
2. 更适合正式项目。
3. 历史变更可追溯。

### 19.3 主要难点

1. 已执行的迁移文件不要随意修改。
2. 文件命名必须规范。
3. 开发期要避免手动改表后忘记写迁移。

## 20. Redis

### 20.1 在项目中的作用

Redis 是可选增强项。

可用于：

1. 在线用户状态。
2. 文档房间状态。
3. 短期对象锁。
4. WebSocket 多实例扩展。
5. 热点数据缓存。

### 20.2 优势

1. 内存读写快。
2. 适合短生命周期状态。
3. 支持过期时间。
4. 适合协同锁和在线状态。

### 20.3 主要难点

1. Redis 中的数据不是主数据源。
2. 需要处理过期和清理。
3. 单实例 MVP 可以先不用 Redis，避免过度设计。

## 21. Docker Compose

### 21.1 在项目中的作用

Docker Compose 用于本地和云端服务编排。

可管理：

1. PostgreSQL。
2. Redis。
3. Spring Boot 后端。
4. Nginx。

### 21.2 优势

相比手工安装：

1. 环境一致。
2. 启动简单。
3. 便于部署说明。

相比 Kubernetes：

1. 学习和配置成本低。
2. 更适合课程项目。
3. 单机演示足够。

### 21.3 主要难点

1. 数据卷持久化。
2. 容器网络。
3. 环境变量管理。
4. 后端等待数据库启动。

## 22. Nginx

### 22.1 在项目中的作用

Nginx 用于云端统一入口。

负责：

1. 托管前端静态文件。
2. 反向代理 `/api` 到 Spring Boot。
3. 反向代理 `/ws` 到 WebSocket。
4. 后续配置 HTTPS。

### 22.2 优势

1. 轻量稳定。
2. 适合静态资源服务。
3. WebSocket 代理成熟。
4. 云服务器部署常用。

### 22.3 主要难点

1. Vue Router history 模式需要 fallback。
2. WebSocket 需要配置 Upgrade 头。
3. API 和前端路径不要冲突。

## 23. 华为云 ECS

### 23.1 在项目中的作用

ECS 是项目云端运行环境。

部署：

1. 前端静态文件。
2. Spring Boot 后端。
3. PostgreSQL 或连接 RDS。
4. Redis 或连接 DCS。
5. Nginx。

### 23.2 优势

1. 课程要求贴合华为云。
2. 部署方式直观。
3. 比 CCE/Kubernetes 更容易上手。
4. 适合演示。

### 23.3 主要难点

1. 安全组配置。
2. Nginx 配置。
3. Docker 持久化。
4. WebSocket 公网访问。
5. 云资源费用控制。

## 24. 华为云 OBS

### 24.1 在项目中的作用

OBS 是增强项，用于对象存储。

可保存：

1. 大模型快照。
2. 导出的 JSON、STL、glTF。
3. 演示文件。
4. 用户上传附件。

### 24.2 优势

相比直接存在 ECS 本地磁盘：

1. 更可靠。
2. 更适合大文件。
3. 与计算服务解耦。

### 24.3 主要难点

1. AK/SK 安全管理。
2. 桶权限配置。
3. SDK 接入。
4. 本地存储和 OBS 存储需要统一抽象。

## 25. FreeCAD

### 25.1 在项目中的作用

FreeCAD 是参考项目，不是直接依赖。

主要参考：

1. CAD 工作台概念。
2. Sketcher 草图思想。
3. PartDesign 特征建模思想。
4. Feature Tree 的建模历史思想。

### 25.2 优势

1. 开源。
2. 基于 OpenCASCADE。
3. 适合理解真实 CAD 系统结构。

### 25.3 主要难点

1. 代码规模大。
2. C++/Python 混合架构与本项目技术栈差异大。
3. 不适合短期直接复用。

## 26. OpenCASCADE / OpenCascade.js

### 26.1 在项目中的作用

OpenCASCADE 是后续几何内核增强方向，OpenCascade.js 是本项目在 Web 端接入 OCCT 的优先验证方案。

MVP 中：

1. 不强依赖。
2. 只预留几何内核接口。
3. 用 Three.js 参数化几何完成演示。

增强阶段：

1. 优先验证 `opencascade.js@beta`，而不是从零编译 OCCT。
2. 在浏览器端执行 B-Rep 建模和 Boolean Cut。
3. 将结果导出为 GLB/glTF 或转换为 Three.js 可渲染 mesh。
4. 支持更真实的拉伸、布尔、倒角和格式交换。

### 26.2 优势

1. 工业级几何建模能力。
2. 支持 B-Rep。
3. 支持 STEP/IGES 等格式。
4. FreeCAD 等项目已有实践基础。
5. OpenCascade.js 已提供 JavaScript/WebAssembly 封装和官方示例，验证路径比自编译 OCCT 更明确。

### 26.3 主要难点

1. Wasm 编译复杂。
2. 前端调用和内存管理复杂。
3. CAD 拓扑命名和特征依赖复杂。
4. 学习成本高，不适合作为首版阻塞项。
5. OpenCascade.js 的 Wasm 文件体积、首次加载、浏览器内存占用和 Vite/Vue bundler 配置需要单独验证。
6. API 风格接近 OCCT C++，不能直接散落在 Vue 组件中，必须封装为 `OpenCascadeJsGeometryKernel`。

## 27. 技术栈总体结论

本项目当前技术路线是合理的：

1. Vue 3 负责快速构建复杂 CAD 工作台界面。
2. Three.js 负责浏览器端图形渲染。
3. Spring Boot 负责稳定后端业务能力。
4. Spring Security 负责鉴权。
5. Spring WebSocket + STOMP 负责实时协同。
6. PostgreSQL 负责结构化数据和模型 JSON 快照。
7. Flyway 保证数据库结构可演进。
8. Docker Compose + Nginx + 华为云 ECS 保证可部署演示。

实现时应坚持一个原则：先让主链路跑通，再逐步增强专业 CAD 能力。

主链路是：

登录 -> 创建项目 -> 进入工作台 -> 绘制草图 -> 生成 3D -> 保存版本 -> 两人协同 -> 云端演示。
