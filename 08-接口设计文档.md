# 基于云的协同机械 CAD 系统接口设计文档

## 1. 文档说明

### 1.1 文档目的

本文档是“基于云的协同机械 CAD 系统”的第 08 阶段产物，用于定义前端、后端、WebSocket 协同层之间的接口契约。

开发人员阅读本文后，应能够明确：

1. 前端通过哪些 REST API 调用后端业务能力。
2. 每个接口的请求方法、路径、权限、请求体、响应体和错误码。
3. WebSocket/STOMP 如何连接、订阅、发送和广播消息。
4. CAD 文档快照、协同 operation、远程光标、在线用户和冲突提示的数据结构。
5. 前端 `api` 目录和后端 Controller/DTO 应如何落地。
6. Postman/Apifox 与浏览器联调时应按什么顺序测试。

### 1.2 设计依据

本文档依据：

1. `02-需求分析文档.md`
2. `03-概要设计文档.md`
3. `05-系统架构设计文档.md`
4. `06-详细设计文档.md`
5. `07-数据库与数据模型设计文档.md`

### 1.3 接口设计范围

本文档覆盖：

1. 认证接口。
2. 项目接口。
3. 项目成员与权限接口。
4. CAD 文档接口。
5. 版本接口。
6. 运行状态接口。
7. WebSocket/STOMP 协同接口。
8. 通用响应格式、错误码和权限规则。

本文档不覆盖：

1. OBS 文件上传签名接口，后续增强阶段再设计。
2. STL/glTF/STEP 导出接口，P1/P2 阶段再扩展。
3. OpenCascade.js 技术验证接口，暂不进入 MVP 主链路。
4. 复杂管理员后台接口。

### 1.4 统一约定

后端基础地址：

```text
本地开发：http://localhost:8080
云端部署：https://your-domain.com
```

REST API 前缀：

```text
/api
```

WebSocket endpoint：

```text
/ws
```

时间格式：

```text
ISO 8601 UTC，例如 2026-06-17T00:00:00Z
```

请求和响应格式：

```text
Content-Type: application/json
```

鉴权方式：

```http
Authorization: Bearer <jwt_token>
```

## 2. 通用接口规范

### 2.1 统一响应结构

所有业务 REST API 使用统一响应包装。

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "data": {}
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `success` | boolean | 是否成功 |
| `code` | string | 业务码 |
| `message` | string | 给前端显示或调试的消息 |
| `data` | any | 业务数据 |

失败示例：

```json
{
  "success": false,
  "code": "PROJECT_NOT_FOUND",
  "message": "项目不存在",
  "data": null
}
```

### 2.2 分页响应结构

MVP 中项目和版本数量不会很大，可以先返回数组。但为了后续扩展，分页响应格式预留如下：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 0
}
```

MVP 采用：

1. 项目列表：直接返回数组。
2. 文档列表：直接返回数组。
3. 成员列表：直接返回数组。
4. 版本列表：直接返回数组。

### 2.3 HTTP 状态码规范

| 状态码 | 说明 | 使用场景 |
| --- | --- | --- |
| 200 | 请求成功 | 查询、修改、保存成功 |
| 201 | 创建成功 | 注册、创建项目、创建文档 |
| 400 | 请求参数错误 | 参数缺失、格式错误 |
| 401 | 未认证 | token 缺失或无效 |
| 403 | 无权限 | 无项目或文档权限 |
| 404 | 资源不存在 | 项目、文档、版本不存在 |
| 409 | 冲突 | 保存版本冲突、成员重复 |
| 500 | 服务端错误 | 未预期异常 |

### 2.4 通用错误码

| 错误码 | HTTP 状态 | 说明 |
| --- | --- | --- |
| `OK` | 200 | 成功 |
| `VALIDATION_ERROR` | 400 | 参数校验失败 |
| `UNAUTHORIZED` | 401 | 未登录或 token 无效 |
| `FORBIDDEN` | 403 | 无权限 |
| `RESOURCE_NOT_FOUND` | 404 | 资源不存在 |
| `CONFLICT` | 409 | 资源冲突 |
| `INTERNAL_ERROR` | 500 | 服务端错误 |

### 2.5 业务错误码

| 错误码 | HTTP 状态 | 说明 |
| --- | --- | --- |
| `USERNAME_ALREADY_EXISTS` | 409 | 用户名已存在 |
| `EMAIL_ALREADY_EXISTS` | 409 | 邮箱已存在 |
| `INVALID_CREDENTIALS` | 401 | 用户名或密码错误 |
| `PROJECT_NOT_FOUND` | 404 | 项目不存在 |
| `DOCUMENT_NOT_FOUND` | 404 | 文档不存在 |
| `VERSION_NOT_FOUND` | 404 | 版本不存在 |
| `PROJECT_MEMBER_NOT_FOUND` | 404 | 项目成员不存在 |
| `PROJECT_MEMBER_ALREADY_EXISTS` | 409 | 成员已存在 |
| `OWNER_ROLE_REQUIRED` | 403 | 需要 OWNER 权限 |
| `EDITOR_ROLE_REQUIRED` | 403 | 需要 OWNER 或 EDITOR 权限 |
| `DOCUMENT_VERSION_CONFLICT` | 409 | 文档版本冲突 |
| `INVALID_CAD_SNAPSHOT` | 400 | CAD 快照格式错误 |
| `INVALID_OPERATION_TYPE` | 400 | operation 类型不合法 |
| `WEBSOCKET_AUTH_FAILED` | 401 | WebSocket 鉴权失败 |
| `COLLABORATION_CONFLICT` | 409 | 协同操作冲突 |

## 3. 认证接口

### 3.1 用户注册

```http
POST /api/auth/register
```

权限：

```text
无需登录
```

请求体：

```json
{
  "username": "alice",
  "email": "alice@example.com",
  "password": "123456"
}
```

校验规则：

| 字段 | 规则 |
| --- | --- |
| `username` | 必填，3 到 64 个字符，唯一 |
| `email` | 必填，邮箱格式，唯一 |
| `password` | 必填，6 到 64 个字符 |

成功响应：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "data": {
    "token": "jwt-token",
    "user": {
      "id": 1,
      "username": "alice",
      "email": "alice@example.com",
      "displayName": "Alice"
    }
  }
}
```

可能错误：

| 错误码 | 说明 |
| --- | --- |
| `VALIDATION_ERROR` | 参数错误 |
| `USERNAME_ALREADY_EXISTS` | 用户名已存在 |
| `EMAIL_ALREADY_EXISTS` | 邮箱已存在 |

后端处理：

1. 校验参数。
2. 校验用户名和邮箱唯一。
3. BCrypt 加密密码。
4. 保存用户。
5. 生成 JWT。
6. 写入 `system_events`，事件类型为 `AUTH_REGISTER`。

可能错误：

后端处理：

1. 查询用户。
2. 校验密码。

4. 写入 `system_events`，事件类型为 `AUTH_LOGIN`。

```http
POST /api/projects/{projectId}/members
```

权限：

```text
OWNER
```

请求体：

```json
{
  "email": "bob@example.com",
```

| 字段 | 规则 |
| --- | --- |

| `role` | 必填，`EDITOR` 或 `VIEWER` |

1. MVP 不提供复杂邀请流程。
2. OWNER 通过输入已注册用户邮箱添加成员。
3. 不允许通过此接口添加第二个 OWNER，若需要转让项目后续再做。

```json
    "user": {

      "displayName": "Bob"
    },
    "role": "EDITOR",
    "createdAt": "2026-06-17T00:00:00Z"
  }
}
```

可能错误：

| 错误码 | 说明 |
| --- | --- |
| `OWNER_ROLE_REQUIRED` | 当前用户不是 OWNER |
| `RESOURCE_NOT_FOUND` | 邮箱对应用户不存在 |
| `PROJECT_MEMBER_ALREADY_EXISTS` | 用户已经是项目成员 |

3. 校验 `baseVersion == currentVersion`。
4. 写入 `document_versions`。
5. 更新 `documents.snapshot_json`、`current_version_number`、`current_version_id`。
6. 写入 `operation_logs` 和 `system_events`。
7. 广播 WebSocket 系统消息 `document.saved`。

可能错误：

| 错误码 | 说明 |
| --- | --- |
| `EDITOR_ROLE_REQUIRED` | 当前用户无编辑权限 |
| `DOCUMENT_VERSION_CONFLICT` | baseVersion 落后 |
| `INVALID_CAD_SNAPSHOT` | 快照结构不合法 |

409 示例：

```json
{
  "success": false,
  "code": "DOCUMENT_VERSION_CONFLICT",
  "message": "文档版本已更新，请刷新后再保存",
  "data": {
    "clientBaseVersion": 3,
    "serverVersion": 4
  }
}
```

## 9. WebSocket/STOMP 接口

### 9.1 连接方式

WebSocket endpoint：

```text
/ws
```

STOMP CONNECT header：

```text
Authorization: Bearer <jwt_token>
```

连接流程：

```text
前端创建 STOMP client
  -> CONNECT /ws
  -> header 携带 JWT
  -> 后端 WebSocketAuthChannelInterceptor 校验
  -> 连接成功后订阅 document topic
  -> 发送 join 消息
```

鉴权要求：

1. CONNECT 阶段必须校验 token。
2. 每条业务消息必须从 Principal 获取当前用户。
3. 不信任客户端消息里的 `userId`。

### 9.2 发送目的地

| 目的地 | 说明 | 权限 |
| --- | --- | --- |
| `/app/documents/{documentId}/join` | 加入文档协同房间 | OWNER/EDITOR/VIEWER |
| `/app/documents/{documentId}/leave` | 离开文档协同房间 | OWNER/EDITOR/VIEWER |
| `/app/documents/{documentId}/cursor` | 发送光标位置 | OWNER/EDITOR/VIEWER |
| `/app/documents/{documentId}/operations` | 发送 CAD 操作 | OWNER/EDITOR |

说明：

1. 协同操作统一使用复数 `/operations`。
2. 光标可以由 VIEWER 发送。
3. 建模 operation 只有 OWNER/EDITOR 可发送。

### 9.3 订阅主题

| Topic | 说明 |
| --- | --- |
| `/topic/documents/{documentId}/presence` | 在线用户广播 |
| `/topic/documents/{documentId}/cursor` | 远程光标广播 |
| `/topic/documents/{documentId}/operations` | CAD 操作广播 |
| `/topic/documents/{documentId}/system` | 系统提示、保存、恢复、冲突、权限变化 |

## 10. CAD 数据接口契约

### 10.1 CAD 快照结构

保存文档和读取文档时使用相同快照结构。

```json
{
  "schemaVersion": "1.0",
  "documentId": "1",
  "name": "Demo Part",
  "unit": "mm",
  "metadata": {
    "currentVersion": 3,
    "updatedAt": "2026-06-17T00:00:00Z"
  },
  "sketches": [
    {
      "id": "sketch-001",
      "name": "Sketch 1",
      "plane": "XY",
      "entities": [],
      "constraints": []
    }
  ],
  "features": []
}
```

校验规则：

1. `schemaVersion` 必须存在，MVP 为 `1.0`。
2. `unit` 必须为 `mm`。
3. `sketches` 必须是数组。
4. `features` 必须是数组。
5. 快照大小建议不超过 5 MB。

## 15. 联调顺序

### 15.1 REST API 联调顺序

建议顺序：

1. `POST /api/auth/register`
2. `POST /api/auth/login`
3. `GET /api/auth/me`
4. `POST /api/projects`
5. `GET /api/projects`
6. `GET /api/projects/{projectId}`
7. `GET /api/projects/{projectId}/members`
8. `POST /api/projects/{projectId}/members`
9. `GET /api/projects/{projectId}/documents`
10. `GET /api/documents/{documentId}`
11. `PUT /api/documents/{documentId}/save`
12. `GET /api/documents/{documentId}/versions`
13. `GET /api/documents/{documentId}/versions/{versionId}`
14. `POST /api/documents/{documentId}/versions/{versionId}/restore`

### 15.2 WebSocket 联调顺序

建议顺序：

1. 浏览器 A 登录并打开工作台。
2. 浏览器 A 连接 `/ws`。
3. 浏览器 A 订阅四类 topic。
4. 浏览器 A 发送 join。
5. 浏览器 B 登录并打开同一文档。
6. 浏览器 B 发送 join。
7. 双方收到在线用户列表。
8. 浏览器 A 发送 cursor。
9. 浏览器 B 收到 cursor。
10. 浏览器 A 创建草图实体并发送 operation。
11. 浏览器 B 收到 operation 并应用。
12. 浏览器 A 保存文档。
13. 浏览器 B 收到 document.saved 系统消息。
14. 双方同时修改同一对象，验证 conflict.warning。

## 17. 接口版本管理

### 17.1 REST API 版本

MVP 使用：

```text
/api
```

后续若需要破坏性升级，可改为：

```text
/api/v2
```

MVP 阶段不引入 `/api/v1`，避免路径过长。

### 17.2 CAD JSON 版本

CAD 快照使用：

```json
{
  "schemaVersion": "1.0"
}
```

前端加载规则：

1. `schemaVersion === "1.0"` 正常加载。
2. 版本缺失或不支持时提示用户。
3. 后端只做基础校验，不做复杂迁移。

### 17.3 Operation 版本

MVP operation 不单独加协议版本。

若后续需要，可增加：

```json
{
  "protocolVersion": "1.0"
}
```

## 18. 安全与部署注意事项

### 18.1 CORS

本地开发允许：

```text
http://localhost:5173
```

云端部署时只允许正式前端域名。

### 18.2 WebSocket Nginx 配置要求

Nginx 必须支持：


接口不得返回：

1. passwordHash。
2. JWT secret。
3. 数据库连接信息。
4. OBS AK/SK。

后端日志应包含：

1. traceId。
2. userId。
3. projectId。
4. documentId。
5. eventType。
6. result。

但不得记录：

1. 明文密码。
2. token 全文。
3. 大体积 snapshot 全文。

## 20. 接口设计结论

本接口设计将系统接口分为两类：

```text
REST API：用户、项目、成员、文档、版本、保存、恢复
WebSocket/STOMP：在线状态、远程光标、协同 operation、系统消息
```

设计重点是：

1. REST API 负责可靠的业务状态和持久化。
2. WebSocket 负责实时协同和增量消息。
3. 文档快照通过 JSONB 保存。
4. 协同 operation 通过 WebSocket 广播并写入 `operation_logs`。
5. 权限判断全部以服务端为准。
6. `baseVersion` 用于保存冲突和协同冲突提示。

下一阶段应编写 `09-原型设计与交互设计文档.md`，将接口和数据模型进一步映射到登录页、项目列表页、CAD 工作台、版本弹窗、成员管理弹窗和协同状态栏的具体交互中。
