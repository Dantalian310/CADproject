# 基于云的协同机械 CAD 系统详细设计文档

## 1. 文档说明

### 1.1 文档目的

本文档是“基于云的协同机械 CAD 系统”的详细设计文档，对概要设计中的模块进行进一步拆解，说明前端、后端、CAD 核心、协同、权限、日志和异常处理在编码阶段应如何落地。

本文档面向后续编码实现，开发人员阅读后应能够明确：

1. 前端工程应该创建哪些目录、组件、store、核心类和工具函数。
2. CAD 文档、草图、特征、操作命令、Undo/Redo 应如何组织。
3. Three.js 渲染层和几何生成层如何隔离。
4. 后端各模块应创建哪些 Controller、Service、Repository、DTO 和 Entity。
5. WebSocket 协同消息如何进入、校验、广播和落库。
6. MVP 阶段哪些逻辑必须实现，哪些逻辑只预留接口。

### 1.2 文档边界

本文档属于第 06 阶段产物，关注“模块内部如何设计”。它与后续文件的边界如下：

| 文件 | 关注重点 | 与本文档关系 |
| --- | --- | --- |
| `07-数据库与数据模型设计文档.md` | 表结构、字段、索引、迁移脚本 | 本文档只定义核心对象和数据流，不展开完整 DDL |

### 1.4 MVP 实现边界

MVP 必须覆盖：

1. 用户注册、登录、当前用户查询。
2. 项目创建、项目列表、项目成员和基础权限。
3. CAD 工作台布局：工具栏、特征树、3D 视图区、属性面板、在线用户、状态栏。
4. 2D 草图：线段、矩形、圆，固定、水平、垂直、尺寸驱动。
5. 3D 建模：拉伸、Cut、布尔并集、布尔差集、参数化立方体/长方体、球体、锥体。
6. 文档保存、加载、版本列表、版本恢复。
7. Undo/Redo。
8. WebSocket 协同：房间加入、在线用户、远程光标、增量操作广播、冲突提示。
9. 健康检查、操作日志和系统事件。

MVP 不强制实现：

1. 工业级 B-Rep。
2. 完整约束求解器。
3. STEP/IGES/DWG 导入导出。
4. 工程图、装配、仿真。
5. Yjs/CRDT。
6. OpenCascade.js 主链路集成。

OpenCascade.js/OCCT Wasm 仅作为 P1/P2 技术验证，首版通过 `GeometryKernel` 抽象预留。

## 3. 前端详细设计

### 3.1 路由设计

路由表：

| 路径 | 页面 | 是否登录 | 说明 |
| --- | --- | --- | --- |
| `/login` | `LoginPage` | 否 | 用户登录 |
| `/register` | `RegisterPage` | 否 | 用户注册 |
| `/projects` | `ProjectListPage` | 是 | 项目列表 |
| `/workspace/:projectId/:documentId` | `WorkspacePage` | 是 | CAD 工作台 |

路由守卫：

1. 进入需要登录页面时检查 `authStore.token`。
2. token 不存在时跳转 `/login`。
3. token 存在但当前用户为空时调用 `/api/auth/me`。
4. 工作台路由进入后加载项目、文档和协同连接。

`guards.ts` 建议职责：

```ts
router.beforeEach(async (to) => {
  if (!to.meta.requiresAuth) return true
  const authStore = useAuthStore()
  if (!authStore.token) return '/login'
  if (!authStore.currentUser) {
    await authStore.loadCurrentUser()
  }
  return true
})
```

参数修改要求：

1. 不直接改对象深层属性。

3. 修改后生成 operation，可用于协同广播。

### 3.3 Pinia Store 详细设计

#### 3.3.1 `auth.store.ts`

状态：

```ts
interface AuthState {
  token: string | null
  currentUser: UserDTO | null
  loading: boolean
}
```

Actions：

1. `login(username, password)`
2. `register(payload)`
3. `loadCurrentUser()`
4. `logout()`

设计要求：

1. token 保存到 `localStorage`。
2. `http.ts` 自动在请求头附加 token。
3. 401 响应时清理 token 并跳转登录页。

#### 3.3.2 `project.store.ts`

状态：

```ts
interface ProjectState {
  projects: ProjectDTO[]
  currentProject: ProjectDTO | null
  members: ProjectMemberDTO[]
  loading: boolean
}
```

Actions：

1. `loadProjects()`
2. `createProject(payload)`
3. `loadProject(projectId)`
4. `loadMembers(projectId)`
5. `addMember(projectId, payload)`
6. `updateMemberRole(projectId, memberId, role)`
7. `removeMember(projectId, memberId)`

#### 3.3.3 `cad.store.ts`

状态：

```ts
interface CadState {
  document: CadDocument | null
  selection: CadSelection | null
  activeTool: CadToolType
  preview: CadPreview | null
  history: HistoryStack
  dirty: boolean
  currentVersion: number
}
```

核心 Actions：

1. `loadDocument(documentId)`
2. `setActiveTool(tool)`
3. `executeCommand(command)`
4. `applyOperation(operation, source)`
5. `undo()`
6. `redo()`
7. `saveDocument()`
8. `rebuildGeometry()`

`executeCommand` 流程：

```text
command.execute(document)
  -> push history undo stack
  -> clear redo stack
  -> mark dirty
  -> rebuild geometry
  -> convert command to operation
  -> collaborationStore.sendOperation(operation)
```

注意：

1. 远程 operation 进入时不再广播。
2. 保存成功后 `dirty = false`。
3. Undo/Redo 也应生成 operation，保证协同用户同步。

#### 3.3.4 `collaboration.store.ts`

状态：

```ts
interface CollaborationState {
  connected: boolean
  documentId: string | null
  onlineUsers: OnlineUser[]
  remoteCursors: Record<string, RemoteCursor>
  lastConflict: ConflictMessage | null
}
```

Actions：

1. `connect(documentId, token)`
2. `disconnect()`
3. `joinDocument(documentId)`
4. `leaveDocument(documentId)`
5. `sendOperation(operation)`
6. `sendCursor(cursor)`
7. `handleMessage(message)`

设计要求：

1. 光标消息节流，建议 50ms 到 100ms 一次。
2. 操作消息不能节流，必须按顺序发送。
3. 收到远程 operation 后调用 `cadStore.applyOperation(operation, 'remote')`。
4. 收到 conflict 时更新 `lastConflict` 并提示用户。

#### 3.3.5 `status.store.ts`

状态：

```ts
interface StatusState {
  apiStatus: 'unknown' | 'ok' | 'error'
  websocketStatus: 'disconnected' | 'connecting' | 'connected' | 'error'
  saveStatus: 'saved' | 'dirty' | 'saving' | 'failed'
  lastError: string | null
}
```

Actions：

1. `setApiStatus(status)`
2. `setWebsocketStatus(status)`
3. `setSaveStatus(status)`
4. `reportError(message)`
5. `clearError()`

## 4. CAD 前端领域模型详细设计

### 4.1 CAD 文档模型

`CadDocument` 是前端 CAD 状态的根对象。

```ts
interface CadDocument {
  schemaVersion: '1.0'
  documentId: string
  name: string
  unit: 'mm'
  sketches: Sketch[]
  features: Feature[]
  metadata: DocumentMetadata
}

interface DocumentMetadata {
  createdAt?: string
  updatedAt?: string
  currentVersion?: number
}
```

设计要求：

1. 所有草图和特征必须有稳定 id。
2. 不保存 Three.js Mesh 到 `CadDocument`。
3. `CadDocument` 必须能序列化成 JSON。
4. 前端保存时提交完整 snapshot，协同时提交增量 operation。

### 4.2 草图模型

```ts
interface Sketch {
  id: string
  name: string
  plane: SketchPlane
  entities: SketchEntity[]
  constraints: SketchConstraint[]
}

type SketchPlane = 'XY'
```

MVP 只支持 XY 平面。后续可扩展 YZ、XZ、自定义基准面。

### 4.3 草图实体模型

```ts
type SketchEntity =
  | LineEntity
  | RectangleEntity
  | CircleEntity
  | EllipseEntity
  | PolylineEntity
  | ConstructionLineEntity

interface BaseSketchEntity {
  id: string
  type: string
  name: string
  visible: boolean
  locked?: boolean
}
```

线段：

```ts
interface LineEntity extends BaseSketchEntity {
  type: 'line'
  start: Point2
  end: Point2
}
```

矩形：

```ts
interface RectangleEntity extends BaseSketchEntity {
  type: 'rectangle'
  origin: Point2
  width: number
  height: number
}
```

圆：

```ts
interface CircleEntity extends BaseSketchEntity {
  type: 'circle'
  center: Point2
  radius: number
}
```

P1 扩展：

```ts
interface EllipseEntity extends BaseSketchEntity {
  type: 'ellipse'
  center: Point2
  radiusX: number
  radiusY: number
}

interface PolylineEntity extends BaseSketchEntity {
  type: 'polyline'
  points: Point2[]
  closed: boolean
}

interface ConstructionLineEntity extends BaseSketchEntity {
  type: 'construction-line'
  start: Point2
  end: Point2
}
```

点类型：

```ts
interface Point2 {
  x: number
  y: number
}
```

### 4.4 约束模型

```ts
type SketchConstraint =
  | FixedConstraint
  | HorizontalConstraint
  | VerticalConstraint
  | DimensionConstraint
  | CoincidentConstraint
```

固定约束：

```ts
interface FixedConstraint {
  id: string
  type: 'fixed'
  entityId: string
}
```

水平约束：

```ts
interface HorizontalConstraint {
  id: string
  type: 'horizontal'
  entityId: string
}
```

垂直约束：

```ts
interface VerticalConstraint {
  id: string
  type: 'vertical'
  entityId: string
}
```

尺寸约束：

```ts
interface DimensionConstraint {
  id: string
  type: 'dimension'
  entityId: string
  dimension: 'width' | 'height' | 'radius' | 'length'
  value: number
}
```

重合约束：

```ts
interface CoincidentConstraint {
  id: string
  type: 'coincident'
  entityIdA: string
  pointKeyA: string
  entityIdB: string
  pointKeyB: string
}
```

MVP 约束求解策略：

1. 不做通用求解器。
2. 添加水平约束时，将线段两端 y 调成一致。
3. 添加垂直约束时，将线段两端 x 调成一致。
4. 添加固定约束时，属性面板禁止编辑该实体坐标。
5. 修改尺寸约束时，直接修改实体参数。
6. 重合约束作为 P1，可通过同步两个端点坐标实现。

### 4.5 特征模型

```ts
type Feature =
  | ExtrudeFeature
  | CutFeature
  | BooleanFeature
  | BoxFeature
  | SphereFeature
  | ConeFeature

interface BaseFeature {
  id: string
  type: string

  suppressed: boolean
  createdAt?: string
}
```

参数化基础体：

```ts
interface Point3 {
  x: number
  y: number

interface BoxFeature extends BaseFeature {
  type: 'box'
  position: Point3
  length: number

  position: Point3
  radius: number

interface ConeFeature extends BaseFeature {
  type: 'cone'
  position: Point3
  baseRadius: number
  height: number
}
```

拉伸特征：

```ts
interface ExtrudeFeature extends BaseFeature {
  type: 'extrude'
  sourceSketchId: string
  sourceEntityId: string
  depth: number
  operation: 'new'

```

切除特征：

```ts
interface CutFeature extends BaseFeature {
  type: 'cut'
  targetFeatureId: string
  toolSketchId: string
  toolEntityId: string

```

布尔特征：

```ts
interface BooleanFeature extends BaseFeature {
  type: 'boolean'
  operation: 'union' | 'difference'
  targetFeatureId: string
  toolFeatureId: string
}
```

设计要求：

1. 特征只引用草图或其他特征 id，不直接引用 mesh。
2. 删除草图实体前要检查是否被特征引用。
3. 重建几何时按 `features` 数组顺序执行。
4. 特征禁用 `suppressed = true` 后不参与几何生成。
5. `box`、`sphere`、`cone` 是直接参数化特征，不依赖草图实体；其几何由位置和尺寸参数重建。

### 4.6 选择模型

```ts
type CadSelection =
  | { kind: 'document'; documentId: string }
  | { kind: 'sketch'; sketchId: string }
  | { kind: 'sketch-entity'; sketchId: string; entityId: string }
  | { kind: 'feature'; featureId: string }
```

选择设计要求：

1. 全局只维护一个主选择。
2. 多选作为 P1。
3. 选中状态不写入 `CadDocument`，只存在前端 UI 状态。
4. 协同中暂不广播选择，只广播光标和操作。

## 5. CAD 命令与 Undo/Redo 详细设计

### 5.1 Command 接口

```ts
interface CadCommand {
  id: string
  type: CadCommandType
  execute(document: CadDocument): CadDocument
  undo(document: CadDocument): CadDocument
  toOperation(): CadOperation
}
```

设计要求：

1. command 必须可撤销。
2. command 执行后必须能转成 operation。
3. command 不直接调用 API。
4. command 不直接操作 Three.js。

### 5.2 Command 类型

MVP command：

1. `CreateSketchEntityCommand`
2. `UpdateSketchEntityCommand`
3. `DeleteSketchEntityCommand`
4. `AddConstraintCommand`
5. `RemoveConstraintCommand`
6. `CreateExtrudeFeatureCommand`
7. `CreateCutFeatureCommand`
8. `CreateBooleanFeatureCommand`
9. `CreatePrimitiveFeatureCommand`
10. `UpdateFeatureCommand`
11. `DeleteFeatureCommand`

### 5.3 HistoryStack

```ts
class HistoryStack {
  private undoStack: CadCommand[] = []
  private redoStack: CadCommand[] = []

  push(command: CadCommand): void
  undo(document: CadDocument): CadDocument
  redo(document: CadDocument): CadDocument
  canUndo(): boolean
  canRedo(): boolean
  clear(): void
}
```

Undo 流程：

```text
点击 Undo
  -> pop undoStack
  -> command.undo(document)
  -> push redoStack
  -> rebuildGeometry
  -> 生成 undo operation
  -> 发送协同消息
```

Redo 流程：

```text
点击 Redo
  -> pop redoStack
  -> command.execute(document)
  -> push undoStack
  -> rebuildGeometry
  -> 生成 redo operation
  -> 发送协同消息
```

### 5.4 远程操作与本地历史

远程 operation 应用规则：

1. 远程 operation 直接修改文档，不进入本地 undoStack。
2. 远程 operation 应记录到操作日志。
3. 如果远程 operation 修改了当前选中对象，则属性面板应刷新。
4. 如果远程 operation 与本地未保存修改冲突，则显示冲突提示。

MVP 冲突策略：

1. 如果两个用户修改不同对象，直接合并。
2. 如果两个用户修改同一 `targetId` 且 `baseVersion` 不一致，显示冲突提示。
3. MVP 不自动合并复杂冲突。
4. 用户可通过刷新文档或保存新版本解决。

## 6. 几何生成与 Three.js 详细设计

### 6.1 GeometryKernel 接口

```ts
interface GeometryKernel {
  buildDocument(document: CadDocument): MeshDescriptor[]
  buildExtrude(feature: ExtrudeFeature, document: CadDocument): MeshDescriptor
  buildCut(feature: CutFeature, document: CadDocument): MeshDescriptor
  buildBoolean(feature: BooleanFeature, document: CadDocument): MeshDescriptor
  buildBox(feature: BoxFeature): MeshDescriptor
  buildSphere(feature: SphereFeature): MeshDescriptor
  buildCone(feature: ConeFeature): MeshDescriptor
}
```

`MeshDescriptor`：

```ts
interface MeshDescriptor {
  id: string
  featureId: string
  geometry: THREE.BufferGeometry
  material: THREE.Material
  transform?: THREE.Matrix4
}
```

设计要求：

1. Vue 组件只能调用 `SceneManager.rebuild(document)`。
2. `SceneManager` 内部调用 `GeometryKernel`。
3. Three.js mesh 不写入 Pinia 文档状态。
4. 后续 `OpenCascadeJsGeometryKernel` 必须实现同一接口。

### 6.2 ThreeGeometryKernel

职责：

1. 将矩形拉伸为盒体或 ExtrudeGeometry。
2. 将圆拉伸为圆柱体。
3. 将简单多边形拉伸为 ExtrudeGeometry。
4. 为草图线条生成辅助显示对象。
5. 将参数化 `box` 特征生成为 `BoxGeometry(length, width, height)`。
6. 将参数化 `sphere` 特征生成为 `SphereGeometry(radius)`。
7. 将参数化 `cone` 特征生成为 `ConeGeometry(baseRadius, height)`。

拉伸规则：

| 草图实体 | 生成方式 |
| --- | --- |
| rectangle | `BoxGeometry(width, height, depth)` 或 `ExtrudeGeometry` |
| circle | `CylinderGeometry(radius, radius, depth)` |
| polyline closed | `Shape + ExtrudeGeometry` |

参数化基础体规则：

| 特征 | 生成方式 | 参数 |
| --- | --- | --- |
| box | `BoxGeometry(length, width, height)` | `position`、`length`、`width`、`height` |
| sphere | `SphereGeometry(radius)` | `position`、`radius` |
| cone | `ConeGeometry(baseRadius, height)` | `position`、`baseRadius`、`height` |

MVP 坐标约定：

1. 草图在 XY 平面。
2. 拉伸方向为 Z 正方向。
3. 单位为 mm，但 Three.js 中按数值直接渲染。
4. 可以用缩放系数改善显示，但数据层仍存 mm。
5. 参数化基础体的 `position` 表示底部参考点，渲染时根据高度或半径平移到 Three.js 几何中心。

### 6.3 CSG 几何生成

`CsgGeometryKernel` 可作为 `ThreeGeometryKernel` 的扩展，内部使用 `three-bvh-csg`。

职责：

1. 实现 Cut。
2. 实现 Boolean Union。
3. 实现 Boolean Difference。

使用边界：

1. 只对封闭 mesh 做布尔。
2. 不承诺工业级拓扑稳定。
3. 布尔失败时给出提示，不破坏原文档。
4. 布尔结果仍作为 feature 渲染结果，不替代特征树。

## 7. 草图交互详细设计

### 7.1 坐标转换

鼠标坐标转换流程：

```text
PointerEvent(clientX, clientY)
  -> 归一化设备坐标 NDC
  -> Raycaster 从 camera 发射射线
  -> 与 XY 草图平面求交
  -> 得到 Point2(x, y)
```

`coordinate.ts`：

```ts
function screenToSketchPoint(
  event: PointerEvent,
  camera: THREE.Camera,
  dom: HTMLElement
): Point2
```

设计要求：

1. 坐标转换统一封装，不在各工具中重复写。
2. 支持网格吸附时，在转换后做 snap。
3. 鼠标预览使用同一坐标体系。

### 7.2 绘制线段

状态机：

```text
idle
  -> pointerDown 设置 start
drawing
  -> pointerMove 更新 preview end
  -> pointerUp 创建 LineEntity
idle
```

生成 command：

```ts
new CreateSketchEntityCommand(sketchId, {
  type: 'line',
  start,
  end
})
```

校验：

1. 线段长度不能为 0。
2. 如果开启水平/垂直辅助吸附，可自动调整 end 坐标。

### 7.3 绘制矩形

状态机：

```text
idle
  -> pointerDown 设置 origin
drawing
  -> pointerMove 更新 width/height preview
  -> pointerUp 创建 RectangleEntity
idle
```

校验：

1. 宽高绝对值不能为 0。
2. 如果用户反向拖动，需要将 origin 调整为左下角或保留 origin + signed width，建议 MVP 统一转正值。

### 7.4 绘制圆

状态机：

```text
idle
  -> pointerDown 设置 center
drawing
  -> pointerMove 更新 radius preview
  -> pointerUp 创建 CircleEntity
idle
```

校验：

1. 半径必须大于 0。
2. 默认半径单位为 mm。

### 7.5 选择与删除

选择流程：

```text
pointerDown
  -> SceneManager.pick
  -> MeshRegistry.findBusinessId
  -> cadStore.setSelection
  -> PropertyPanel 刷新
```

删除流程：

1. 用户按 Delete 或点击删除按钮。
2. 检查选中对象类型。
3. 检查依赖关系。
4. 生成 Delete command。
5. 执行并广播 operation。

依赖校验：

1. 被特征引用的草图实体不允许直接删除。
2. 可以提示“该草图被特征使用，请先删除相关特征”。

### 7.6 参数编辑与尺寸驱动

属性面板修改参数流程：

```text
输入参数
  -> 校验数值
  -> UpdateSketchEntityCommand
  -> apply constraints
  -> rebuild geometry
  -> send operation
```

尺寸驱动规则：

1. 矩形宽高修改后更新 `width`、`height`。
2. 圆半径修改后更新 `radius`。
3. 线段长度修改时保持 start 不变，按原方向调整 end。
4. 水平线长度修改时只改 end.x。
5. 垂直线长度修改时只改 end.y。

1. MVP 阶段 operation 不立即写完整文档快照。
2. 文档快照由保存接口写入。
3. operation log 用于调试、审计和版本解释。

#### 9.7.5 `ConflictService`

MVP 冲突检测：

```java
boolean hasConflict(OperationMessage message, DocumentEntity document) {
    return message.getBaseVersion() < document.getCurrentVersion()
        && recentlyModifiedSameTarget(message.getDocumentId(), message.getTargetId());
}
```

冲突提示内容：

1. documentId。
2. targetId。
3. operationId。
4. reason。
5. serverVersion。

MVP 只提示，不自动解决。

### 9.8 日志与可观测性模块

#### 9.8.2 `SystemEventService`

3. 记录房间加入、离开。
4. 记录权限拒绝。
5. 记录保存失败、版本冲突。

方法：


结构化日志字段：

1. `traceId`
2. `userId`
3. `projectId`

6. `result`

## 11. 核心业务流程详细设计

### 11.1 打开工作台流程

```text
前端进入 /workspace/:projectId/:documentId
  -> authStore 确认登录
  -> projectStore.loadProject(projectId)
  -> cadStore.loadDocument(documentId)
  -> SceneManager.init
  -> SceneManager.rebuild(document)
  -> collaborationStore.connect(documentId)
  -> collaborationStore.joinDocument(documentId)
  -> statusStore 显示 connected
```

失败处理：

1. 文档不存在：跳转项目页。
2. 无权限：提示并跳转项目页。
3. WebSocket 失败：允许单人编辑，但状态栏提示协同不可用。

### 11.2 建模操作流程

```text
用户选择工具
  -> 鼠标在视图区操作
  -> sketchTool 生成 command
  -> cadStore.executeCommand
  -> 更新 CadDocument
  -> rebuildGeometry
  -> command.toOperation
  -> collaborationStore.sendOperation
  -> 后端记录并广播
```

关键点：

1. 本地先执行，保证交互流畅。
2. 后端广播用于同步其他用户。
3. 如果服务端返回冲突，前端提示用户。

### 11.3 保存文档流程

```text
点击保存
  -> statusStore.saveStatus = saving
  -> cadStore 序列化 CadDocument
  -> PUT /api/documents/{documentId}/save
  -> 后端校验权限和 baseVersion
  -> 生成新版本
  -> 返回 currentVersion
  -> 前端更新 currentVersion 和 dirty=false
  -> statusStore.saveStatus = saved
```

保存失败：

1. 401：跳转登录。
2. 403：提示无权限。
3. 409：提示版本冲突。
4. 500：提示保存失败，不清除 dirty。

### 11.4 远程 operation 流程

```text
后端广播 operation
  -> collaborationStore.handleMessage
  -> 判断是否来自当前用户
  -> 来自当前用户则忽略或只确认
  -> 来自其他用户则 cadStore.applyOperation
  -> rebuildGeometry
  -> 更新状态栏
```

注意：

1. 服务端广播中应包含 sender userId。
2. 当前用户自己的 operation 可忽略，避免重复应用。
3. 如果 operation 应用失败，应记录 lastError。

### 11.5 恢复历史版本流程

```text
打开版本历史
  -> GET versions
  -> 选择版本
  -> POST restore
  -> 后端创建新版本
  -> 返回新 document snapshot
  -> 前端替换 CadDocument
  -> 清空 Undo/Redo
  -> rebuildGeometry
  -> 广播 document restored
```

设计要求：

1. 恢复后清空本地历史栈。
2. 恢复动作生成新版本。
3. 协同用户收到恢复消息后刷新文档。

## 12. 权限详细设计

### 12.1 前端权限控制

前端根据 `myRole` 控制 UI：

| 权限 | UI 行为 |
| --- | --- |
| OWNER | 显示成员管理、编辑、保存 |
| EDITOR | 显示编辑、保存，不显示成员管理 |
| VIEWER | 禁用建模工具、保存、Undo/Redo |

注意：

1. 前端权限只提升体验，不能作为安全依据。
2. 后端必须再次校验。

### 12.2 后端权限控制

所有接口需要按资源校验：

| 操作 | 权限 |
| --- | --- |
| 查看项目 | OWNER/EDITOR/VIEWER |
| 创建文档 | OWNER/EDITOR |
| 保存文档 | OWNER/EDITOR |
| 恢复版本 | OWNER/EDITOR |
| 管理成员 | OWNER |
| 发送 cursor | OWNER/EDITOR/VIEWER |
| 发送 operation | OWNER/EDITOR |

权限校验建议由 `ProjectMemberService` 提供统一方法，避免散落在各 Service。

## 13. 数据一致性详细设计

### 13.1 版本号一致性

设计规则：

1. `documents.currentVersion` 是当前服务端版本号。
2. 前端保存时提交 `baseVersion`。
3. 后端保存成功后版本号加 1。
4. `document_versions.versionNumber` 不允许重复。

### 13.2 Operation 与 Snapshot 关系

MVP 采用：

1. operation 用于实时协同。
2. snapshot 用于持久化和版本恢复。
3. operation 不作为恢复文档的唯一依据。

这样可以降低实现难度，避免必须从所有 operation 重放恢复状态。

### 13.3 前端状态一致性

规则：

1. 所有 CAD 修改必须走 command。
2. 所有 command 必须更新 `dirty`。
3. 所有远程 operation 必须通过 `applyRemoteOperation`。
4. 保存成功后更新 `currentVersion`。
5. 恢复历史版本后清空本地历史栈。

## 14. 异常处理详细设计

### 14.1 前端异常处理

HTTP 异常：

| 状态 | 前端处理 |
| --- | --- |
| 400 | 显示业务错误 |
| 401 | 清 token，跳转登录 |
| 403 | 显示无权限 |
| 404 | 显示资源不存在 |
| 409 | 显示版本冲突 |
| 500 | 显示系统错误 |

WebSocket 异常：

1. 连接失败：状态栏显示 `error`。
2. 断开连接：尝试重连。
3. 重连失败：允许本地编辑，但禁止发送协同消息。
4. operation 应用失败：显示同步错误。

### 14.2 后端异常处理

统一由 `GlobalExceptionHandler` 处理：

1. 参数校验异常。
2. 业务异常。
3. 鉴权异常。
4. 未知异常。

未知异常要求：

1. 返回通用错误信息。
2. 后端记录详细日志。
3. 不把堆栈直接返回前端。

## 15. 性能与资源释放详细设计

### 15.1 前端性能

MVP 策略：

1. 文档变更后整体 rebuild。
2. 文档规模控制在课程演示级别。
3. Three.js 对象 rebuild 前统一 dispose。
4. 光标消息节流。
5. 高频拖拽只更新 preview，pointerUp 后生成正式 command。

P1 优化：

1. feature 级局部 rebuild。
2. 大量草图实体做空间索引。
3. 远程 operation 批处理。

### 15.2 后端性能

MVP 策略：

1. 版本列表不返回 snapshot。
2. snapshotJson 使用 PostgreSQL JSONB。
3. operation log 同步写入即可。
4. 在线状态存内存。

P1 优化：

1. Redis 存在线状态。
2. operation log 异步写入。
3. 大 snapshot 存 OBS。

### 15.3 Three.js 资源释放

`SceneManager.disposeObject` 要释放：

1. `BufferGeometry.dispose()`
2. `Material.dispose()`
3. texture dispose，若后续使用贴图。
4. renderer dispose。

否则多次进入工作台可能导致浏览器内存增长。

## 19. 开发约束

### 19.1 前端约束

1. 组件不要直接修改深层 CAD 文档对象。
2. 所有 CAD 修改走 command。
3. 所有 Three.js 对象由 `SceneManager` 管理。
4. 所有 HTTP 调用通过 `api` 目录。
5. 所有协同消息通过 `collaboration` 目录。
6. 不把 OpenCascade.js 放进 MVP 主链路。

### 19.2 后端约束

1. Controller 不写复杂业务。
2. Service 方法明确事务边界。
3. Entity 不直接返回给前端。
4. 权限校验不能只靠前端。
5. WebSocket operation 必须校验权限。
6. 日志记录不能影响主流程成功，必要时可降级。

### 19.3 文档与命名约束

1. 前端类型名使用明确领域名，如 `CadDocument`、`SketchEntity`。
2. 后端 DTO 后缀统一为 `Request`、`Response`、`DTO`。
3. 数据库表名使用小写复数。
4. operation type 使用点分式命名，如 `feature.created`。
5. 文档中的 P0 功能优先实现，P1/P2 不得阻塞主链路。

## 20. 详细设计结论

本详细设计将系统拆解为前端 CAD 领域核心、Three.js 渲染层、Pinia 状态层、REST API、Spring Boot 后端业务层、WebSocket 协同层和可观测性模块。

后续编码时应坚持以下主线：

```text
CadDocument JSON 特征树
  -> Command 修改
  -> GeometryKernel 生成 mesh
  -> SceneManager 渲染
  -> Operation 广播
  -> Snapshot 保存版本
```

这个主线能够同时满足课程实验最低要求、协同演示要求和后续扩展 OpenCascade.js/OCCT Wasm 的架构要求。
